description: 验证原型页面与功能规格文档的匹配度，并在不匹配时提供同步选择。
---

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于验证生成的HTML原型页面是否准确反映了功能规格文档中定义的需求。如果发现不匹配，系统会提示用户选择是更新原型还是更新规格，并自动执行同步操作。

**目标**：
1. 确保原型与规格的一致性
2. 发现需求缺失或理解偏差
3. 提供不匹配时的同步机制
4. 生成验证报告

## 执行流程

### 1. 文件定位

a. **定位规格文档**：用户指定路径 → 从原型README.md提取 → 搜索spec.md文件
b. **定位原型文件**：用户指定路径 → 搜索docs/prototypes/最新版本
c. **文件确认**：显示文件路径，询问用户确认

### 2. 内容提取与解析

#### 2.1 从规格文档提取验证要素

- 功能需求（FR）：需求描述、涉及字段、验收标准
- 用户故事（US）：角色、功能、目的、操作步骤
- 验收场景（AS）：场景描述、操作、预期结果
- 数据字段：字段名、类型、必填性、验证规则
- 页面交互：按钮、操作、跳转、反馈

#### 2.2 从原型页面提取实现要素

- 页面结构：页面文件、页面标题、导航路径
- 表单字段：字段名、类型、必填标识、placeholder
- 数据表格：列定义、数据类型、排序、筛选
- 操作按钮：按钮文本、样式类型（主/次/危险）
- 交互元素：弹窗、确认框、提示信息

### 3. 匹配度验证

对以下维度进行匹配验证：

- **功能需求匹配**：检查每个FR在原型中是否有对应实现
- **用户故事匹配**：检查每个US在原型中是否支持该流程
- **数据字段匹配**：对比规格定义的字段与原型中的字段
- **交互流程匹配**：验证原型中的交互是否符合规格定义

**匹配状态定义**：
- **匹配**：原型完全实现了需求
- **部分匹配**：原型实现了需求，但缺少部分细节
- **不匹配**：原型未实现需求或实现不正确
- **无法判断**：需要人工确认

### 4. 生成验证报告

生成验证报告，包含：
- 基本信息：规格文档、原型路径、原型版本、验证时间、验证结果
- 匹配度统计：各验证维度的匹配情况统计表
- 详细验证结果：各维度的详细匹配表格
- 不匹配项汇总：严重不匹配、一般不匹配、需要人工确认
- 验证结论：根据匹配率给出结论（>=95%通过，85-95%基本匹配，70-85%一般，<70%较低）
- 建议操作：根据验证结果给出建议

**报告保存路径**：`docs/prototypes/{项目名}/{模块名}/{版本号}/validation-report-{时间戳}.md`

### 5. 不匹配处理（核心流程）

**重要提示**：本命令的核心价值在于**明确展示原型与Spec的差异点，并让用户决定以谁为准**。

如果发现不匹配项，**必须**生成静态HTML页面，让用户一次性查看所有差异并提交处理方式。

#### 5.1 收集所有差异项

对每个差异项，收集以下信息：
- **ID**：差异项唯一标识（diff-001, diff-002...）
- **类型**：功能缺失/字段缺失/实现方式不一致/功能增强
- **严重程度**：严重/高/中/低
- **影响范围**：FR编号或功能模块
- **Spec定义**：详细引用规格文档中的原文
- **原型实现**：详细描述原型中的实现情况
- **可能原因**：分析可能的原因
- **推荐处理方式**：系统根据差异类型推荐的处理方式

#### 5.2 生成差异验证表单

**操作**：
1. 将所有差异项整理为JSON格式，包含：
   ```json
   {
     "validationInfo": {
       "specPath": "...",
       "prototypePath": "...",
       "validationTime": "...",
       "matchRate": 78,
       "validationMode": "prototype",
       "resultFilePath": "...",
       "resultFileName": "..."
     },
     "differences": [
       {
         "id": "diff-001",
         "type": "功能缺失",
         "severity": "严重",
         "affectedFR": "FR-001",
         "specDefinition": "...",
         "implementation": "...",
         "possibleReason": "...",
         "recommendedAction": "更新原型"
       }
     ]
   }
   ```

2. 调用脚本生成HTML表单：
   ```bash
   .specify/scripts/bash/generate-difference-validation-form.sh \
     "$VALIDATION_DIR" \
     "$VALIDATION_NAME" \
     "$VALIDATION_DIR/differences.json" \
     "$VALIDATION_DIR/difference-validation-form.html"
   ```

3. 将差异数据保存为JSON文件：`$VALIDATION_DIR/differences.json`
4. 生成HTML表单文件：`$VALIDATION_DIR/difference-validation-form.html`

5. 向用户展示：
   ```
   ════════════════════════════════════════════════════
   检测到 {N} 个差异项
   
   已生成差异验证表单，请按以下步骤操作：
   
   1. 在浏览器中打开以下文件：
      {VALIDATION_DIR}/difference-validation-form.html
   
   2. 查看所有差异项的详细信息（Spec定义、原型实现、可能原因）
   
   3. 为每个差异项选择处理方式：
      - 更新原型（以Spec为准）
      - 更新Spec（以原型为准）
      - 补充其他意见（填写自定义说明）
      - 跳过（暂不处理）
   
   4. 可以使用批量操作快速设置多个差异项的处理方式
   
   5. 所有差异项处理完成后，点击"提交答案"按钮
   
   6. 系统将下载结果JSON文件，请保存到：
      {VALIDATION_DIR}/difference-validation-results.json
   
   7. 保存完成后，请告诉我"已完成"，系统将继续执行同步操作
   
   ════════════════════════════════════════════════════
   ```

6. **等待用户完成表单填写并保存结果文件**

#### 5.3 读取用户选择并执行同步操作

**操作**：
1. 检查结果文件是否存在：`$VALIDATION_DIR/difference-validation-results.json`
2. 如果不存在，等待用户完成表单填写
3. 读取结果文件，解析用户选择
4. **如果用户选择了"更新Spec"**（⚠️ 必须执行版本管理）：
   - **必须先加载并完全遵循**模块文件：`.cursor/modules/spec-version-management.md`
   - 读取当前版本号（从 `FEATURE_DIR/VERSION` 文件，如果不存在则默认为 v1.0.0）
   - 计算新版本号（Minor版本递增，如：v1.0.1 → v1.1.0）
   - 保存旧版本到 `FEATURE_DIR/versions/{旧版本号}-spec.md`
   - 更新 `FEATURE_DIR/VERSION` 文件为新版本号
   - 准备CHANGELOG记录：
     - 变更类型：🖼️ 原型验证调整
     - 变更原因：原型验证阶段发现不匹配项，根据原型实现调整Spec
     - 变更内容：列出所有更新的差异项
     - 影响范围：说明对开发的影响
     - 验证状态：✅ 已通过原型验证
5. 根据用户选择执行同步：
   - **更新原型**：解析HTML文件，定位修改位置，生成新代码片段，插入并保存
   - **更新规格**：
     * 更新spec.md头部版本信息
     * 解析规格文档，定位修改位置，删除或注释字段定义，补充说明
     * 在版本信息章节添加新版本记录
   - **补充其他意见**：记录用户意见到待处理项，不执行自动同步
   - **跳过**：记录到待处理项
6. **如果更新了Spec，更新CHANGELOG.md**：
   - 在 `FEATURE_DIR/CHANGELOG.md` 文件开头（`---` 之后）添加新版本记录

### 6. 生成同步日志

生成同步日志，包含：
- 同步信息：同步时间、规格文档、原型路径、处理模式
- 同步操作记录：不匹配项、处理方式、修改文件、修改内容、状态
- 统计：总不匹配项、更新原型数、更新规格数、跳过数、失败数
- 修改文件清单：原型文件和规格文档的修改清单
- 建议：根据同步结果给出建议

**日志保存路径**：`docs/prototypes/{项目名}/{模块名}/{版本号}/sync-log-{时间戳}.md`

### 7. 重新验证（可选）

同步操作完成后，询问用户是否重新验证。如果重新验证，重复执行步骤3-6。

### 8. 输出最终报告

输出最终报告，包含验证结果、生成文件、下一步操作建议。

## 命令参数说明

```
/原型验证 [选项]

选项:
  --spec=<path>           指定规格文档路径
  --prototype=<path>      指定原型路径
  --report-only           仅生成验证报告，不执行同步操作
  --auto-fix=<strategy>   自动修复策略（prototype/spec/smart）
  --output=<path>         指定报告输出路径

示例:
  /原型验证
  /原型验证 --spec=docs/specs/1-user/spec.md
  /原型验证 --prototype=docs/prototypes/user/v1/
  /原型验证 --report-only
```

## 注意事项

1. **验证时机**：原型初次生成后、规格有重大变更后、产品评审会议前、开始编码前
2. **同步策略**：优先以规格为准，如果原型经过充分评审，可以更新规格
3. **静态原型限制**：动态交互无法在静态HTML中完全展示，用注释说明即可
4. **版本管理**：每次同步操作都会生成同步日志，建议在同步后创建新版本
5. **团队协作**：验证报告可作为评审会议的输入，同步日志可作为变更记录
