## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令支持多种输入方式：
- **文本描述**：`/编写规格 我想添加用户管理功能...`
- **禅道需求链接**：`/编写规格 https://zentao.example.com/story/view/123`
- **在线网页/原型**：`/编写规格 https://example.com/prototype/`
- **HTML文件夹**：`/编写规格 /path/to/prototype/`（支持Axure、Figma等）
- **单个HTML文件**：`/编写规格 ./user-list.html`
- **图片上传**：`/编写规格 [上传UI截图]`

**新增功能：支持更新现有Spec**
- 命令会自动检测是否存在匹配的现有Spec文件
- 如果检测到现有Spec，会询问用户选择"更新现有Spec"或"创建新功能"
- 选择更新时，会自动进行版本管理（创建新版本），保留历史记录
- 更新模式下会合并新内容与现有内容，确保编号连续性

用户在触发消息中在 `/编写规格` 之后键入的文本**就是**输入内容。假设即使 `$ARGUMENTS` 在下面按字面出现，您也始终在此对话中拥有它。除非他们提供了一个空命令，否则不要要求用户重复它。

## 执行流程

### 步骤0：输入识别与预处理

**检测输入类型**（按优先级）：
1. URL 且包含 `zentao` 和 `story`（不区分大小写）→ **禅道需求解析**
2. 以 `http://` 或 `https://` 开头 → **在线网页/原型解析**
3. 路径特征（`/` 或 `./` 或盘符）且文件扩展名为`.doc`、`.docx`或`.pdf`（不区分大小写）→ **文档解析**
4. 路径特征（`/` 或 `./` 或盘符）且指向文件夹且包含.html文件 → **HTML文件夹批量解析**
5. 以`.html`结尾 → **单HTML文件解析**
6. 对话中包含图片 → **图片需求提取**
7. 否则 → **文本描述**（跳过此步骤，直接到步骤1）

**各输入类型的处理**：

- **禅道需求解析**：
  - **⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/禅道需求解析.md`
  - **加载方式**：使用 `read_file` 工具读取该模块文件，理解完整的执行流程
  - **执行要求**：严格按照模块中定义的执行流程执行
  - 输出：`featureDescription`（功能描述文本）、`sourceInfo`（包含URL、Story ID、截图等）
  - 执行：生成一个功能描述 → 继续执行步骤1-7

- **在线网页/原型解析**：
  - **⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/在线网页解析.md`
  - **加载方式**：使用 `read_file` 工具读取该模块文件，理解完整的执行流程
  - **执行要求**：严格按照模块中定义的7个阶段执行，不得跳过任何阶段
  - **关键阶段**：
    * 阶段2.5：页面滚动检查与完整内容提取（⚠️ 必须执行，避免遗漏需求）
    * 阶段5：推断功能需求（优先使用侧边备注中的业务规则）
  - **设计文件特殊处理**（如果检测到URL包含`mastergo.com`或`figma.com`/`figma.io`）：
    * **⚠️ 必须额外执行**：加载并遵循 `.cursor/modules/在线原型深度分析.md`
    * **适用范围**：
      - MasterGo：URL包含 `mastergo.com`
      - Figma：URL包含 `figma.com` 或 `figma.io`
      - 其他设计工具：根据实际情况判断
    * **深度分析要求**（所有设计工具通用）：
      - 完整图层扫描：遍历所有图层（包括折叠和隐藏的图层）
      - UI元素深度提取：统计卡片、筛选器、标签页、视图切换、表格、图表、按钮、表单等所有元素
      - 功能逻辑推断：数据流向、交互流程、业务规则
      - 设计规范提取：颜色、间距、字体、圆角、阴影
      - 完整性验证：使用检查清单确保所有元素都被提取
    * **工具特定适配**：
      - MasterGo：使用MasterGo特定的DOM选择器和图层遍历方式
      - Figma：使用Figma特定的DOM选择器，注意React渲染和权限处理
      - 但分析逻辑和检查清单完全相同
    * **输出增强**：在`web-source.md`中必须包含完整的图层分析报告和UI元素清单，并标注设计工具类型
  - 输出：`featureDescription`、`sourceInfo`、`isBatch`、`batchData`（如果是批量）
  - 执行：单页面 → 继续执行步骤1-7；多页面批量 → 为每个功能执行步骤1-5（跳过步骤6），生成批量报告后结束

- **文档解析**：
  - **⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/文档解析.md`
  - **加载方式**：使用 `read_file` 工具读取该模块文件，理解完整的执行流程
  - **执行要求**：严格按照模块中定义的6个阶段执行，不得跳过任何阶段
  - **支持的文档格式**：
    * `.docx` - Microsoft Word 2007+文档（推荐）
    * `.pdf` - PDF文档
    * `.doc` - Microsoft Word 97-2003文档（支持有限）
  - **关键阶段**：
    * 阶段1：文档解析（提取文本、表格、图片）
    * 阶段2：图片提取与处理（分类识别、可选OCR）
    * 阶段3：内容分析与信息提取（功能模块、用户故事、数据模型等）
    * 阶段4：表格数据处理
    * 阶段5：生成功能描述文本
  - **输出内容**：
    * `featureDescription` - 功能描述文本（Markdown格式）
    * `sourceInfo` - 源文档信息（文档路径、格式、页数、解析时间等）
    * `images` - 提取的图片文件列表（保存到`images/`目录）
    * `errors` - 错误列表（如果有）
    * `warnings` - 警告列表（如果有）
  - **执行**：生成功能描述 → 保存源文档信息到`source-document.md` → 保存图片到`images/`目录 → 继续执行步骤1-7
  - **注意事项**：
    * `.doc`格式支持不完整，优先建议使用`.docx`
    * OCR识别默认关闭，可通过参数启用（耗时较长）
    * 内容提取基于关键词匹配，建议人工验证

- **HTML文件夹批量解析**：
  - 扫描.html文件，识别模块，逐模块解析
  - 执行：为每个模块执行步骤1-5（跳过步骤6），生成批量报告后结束

- **单HTML文件解析**：
  - 读取HTML → 提取UI元素 → 识别页面类型 → 推断功能需求
  - 执行：将提取的需求作为"功能描述" → 继续执行步骤1-7

- **图片需求提取**（如果检测到）：

1. **识别页面类型**：
   - **列表页**：存在表格、分页组件、筛选器 → 查询、筛选、分页、批量操作需求
   - **表单页**：大量输入框、提交按钮 → 数据录入、字段验证、保存需求
   - **详情页**：只读展示、标签-值对 → 数据展示、查看需求
   - **树形页**：树形结构、节点展开 → 树形管理、层级操作需求
   - **数据分析页**：图表、统计指标 → 数据可视化、统计分析需求

2. **提取UI元素**（⚠️ 必须完整提取所有元素）：
   - **统计卡片**（如果存在）：
     - 卡片数量、每个卡片的标题、值（数字、百分比、单位）
     - 变化趋势（上升/下降、变化值）、图标或装饰元素、布局方式
   - **表格**：列头（字段名）、操作列（编辑、删除、详情按钮）、表格行数据、序号列、分页配置（页码、每页条数、快速跳转）、排序和筛选功能
   - **表单**：
     - 输入框类型：文本框、数字框、日期选择器、下拉框、单选框、复选框、文本域
     - 字段标签：中文名称
     - 必填标记：红色星号 `*` 或其他标记
     - 占位符：placeholder提示文本
     - 验证提示：字段下方的说明文字
   - **筛选器**：
     - 筛选字段列表（设备名称、设备类型、状态等）
     - 每个筛选字段的输入类型（输入框、下拉框、日期选择器等）
     - 筛选字段的默认值或占位符
     - 筛选按钮（查询、重置、刷新等）
     - 筛选器的布局方式（行内、块级等）
   - **标签页**（如果存在）：
     - 标签页数量、每个标签页的名称和计数（如"电能表(10)"）
     - 标签页的激活状态、标签页切换的交互逻辑
   - **视图切换**（如果存在）：
     - 视图类型（网格视图、列表视图）、切换按钮的位置和样式、当前激活的视图类型
   - **按钮**：主按钮（提交、保存、确认）、次要按钮（取消、返回、重置）、操作按钮（新增、导出、批量删除）
   - **分页器**：页码、每页条数、快速跳转
   - **导航**：面包屑、侧边导航、Tab标签
   - **图表**（如果存在）：
     - 图表类型（柱状图、折线图、饼图、环形图等）
     - 图表的标题和说明、数据维度（X轴、Y轴、系列）
     - 图表的颜色配置、交互功能（tooltip、缩放、图例等）

3. **提取字段属性**：
   - 字段名称：从标签文本识别
   - 字段类型：根据输入框样式推断（文本/数字/日期/枚举/布尔/文件）
   - 必填性：是否有 `*` 标记
   - 长度限制：从UI大小或提示文字推断
   - 验证规则：从提示文字或错误信息推断（如"请输入正确的邮箱格式"）

4. **提取交互需求**：
   - 按钮点击：提交、保存、删除、导出等操作
   - 页面跳转：新增按钮 → 跳转表单页、编辑按钮 → 跳转编辑页、详情链接 → 跳转详情页
   - 确认对话框：删除操作需要二次确认
   - 加载状态：提交后显示loading
   - 成功/失败提示：操作后显示消息提示

5. **推断功能需求**：
   - 基于页面类型和UI元素，生成结构化的功能需求列表
   - 识别核心实体（如"用户"、"订单"、"商品"）
   - 生成用户故事（如"作为管理员，我想要查询用户列表，以便找到目标用户"）
   - 提取数据字段定义（字段名、类型、必填性、验证规则）

6. **生成功能描述文本**：
   - 汇总所有提取信息，生成完整的文字描述
   - 将提取的需求作为"功能描述"，继续执行步骤1-7

### 步骤0.5：需求明确度评估与模式选择

**⚠️ 重要**：此步骤仅适用于**文本描述**输入类型。对于禅道、网页、HTML、图片等结构化输入，跳过此步骤，直接执行步骤1。

**⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/需求探索模式.md`

**加载方式**：使用 `read_file` 工具读取该模块文件，理解完整的执行流程

**执行要求**：严格按照模块中定义的执行流程执行：
- 执行"步骤0.5：需求明确度评估与模式选择"
- 根据评估结果和用户选择，执行"探索性对话模式"或"引导式澄清模式"
- 模式完成后，继续执行步骤1-7（标准流程）

**模块详细说明**：请参考模块文件 [需求探索模式.md](../modules/需求探索模式.md) 获取完整的执行流程和对话模板，包括：
- 明确度评估维度和评估结果分类
- 模式选择提示和用户选择处理
- 探索性对话模式的三个阶段（需求理解与初步分析、渐进式澄清、需求整理与确认）
- 引导式澄清模式的四个阶段（思路提供与初步分析、关键问题提问、深入提问、需求确认）
- 对话记录格式和规格生成流程

鉴于该功能描述（或从HTML/图片提取的需求，或探索/引导模式完成后的需求），请执行以下操作：

1. **生成简洁的短名称**（2-4 个单词）作为功能标识：
   - 分析功能描述并提取最有意义的关键词
   - 创建一个 2-4 个单词的短名称，捕获功能的本质
   - 尽可能使用动作-名词格式（例如，"add-user-auth"、"fix-payment-bug"）
   - 保留技术术语和首字母缩略词（OAuth2、API、JWT 等）
   - 保持简洁但足够描述以一目了然地理解功能
   - 示例：
     - "我想添加用户身份验证" → "user-auth"
     - "为 API 实施 OAuth2 集成" → "oauth2-api-integration"
     - "为分析创建仪表板" → "analytics-dashboard"
     - "修复支付处理超时错误" → "fix-payment-timeout"

1.5. **检测现有Spec并选择操作模式**（⚠️ 新增：支持更新现有Spec）：

   **检测策略**：
   - 检查 `docs/specs/` 目录下是否存在匹配的现有Spec：
     * 查找所有匹配 `docs/specs/[0-9]+-<short-name>/spec.md` 的目录
     * 如果找到多个匹配，列出所有匹配的Spec供用户选择
     * 如果找到单个匹配，自动使用该Spec
     * 如果未找到匹配，继续执行步骤2（创建新功能）
   
   **用户选择处理**：
   - 如果检测到现有Spec（单个或多个）：
     1. **列出匹配的Spec**（如果多个）：
        ```markdown
        📋 检测到以下现有Spec可能与您的需求相关：
        
        1. `026-cloud-billing-operations-analysis` (v1.2.0) - 云计费运营分析
        2. `022-cross-cloud-management-platform` (v1.0.0) - 跨云管理平台
        
        请选择操作：
        - **更新现有Spec**：输入编号（如：1）或输入完整路径（如：`docs/specs/026-cloud-billing-operations-analysis/spec.md`）
        - **创建新功能**：输入 "new" 或 "创建新功能"
        - **取消**：输入 "cancel" 或 "取消"
        ```
     
     2. **如果用户选择更新现有Spec**：
        - 设置 `$UPDATE_MODE=true`
        - 设置 `$FEATURE_DIR` 为现有Spec的目录路径
        - 设置 `$FEATURE_SPEC` 为现有Spec的文件路径（`$FEATURE_DIR/spec.md`）
        - 读取现有版本号（从 `$FEATURE_DIR/VERSION` 文件）
        - **跳过步骤2**（创建新功能），直接执行步骤3及后续步骤
        - 在步骤4执行前，先执行版本管理（创建新版本，而不是初始化）
     
     3. **如果用户选择创建新功能**：
        - 设置 `$UPDATE_MODE=false`
        - 继续执行步骤2（创建新功能）
     
     4. **如果用户取消**：
        - 终止命令执行
   
   **更新模式下的版本管理**（在步骤4执行前）：
   - **⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/spec-version-management.md`
   - 读取当前版本号：从 `$FEATURE_DIR/VERSION` 文件读取
   - 确定版本类型：
     * 如果用户明确说明是"重大功能变更"或"迭代开发" → `major`（如：v1.3.0 → v2.0.0）
     * 如果用户说明是"功能调整"或"内容补充" → `minor`（如：v1.0.1 → v1.1.0）
     * 如果用户说明是"澄清"或"修正" → `patch`（如：v1.0.0 → v1.0.1）
     * 如果无法确定，默认使用 `minor`
   - 创建新版本：
     * 保存当前Spec到 `versions/` 目录
     * 更新 `VERSION` 文件
     * 更新 `CHANGELOG.md`（变更类型：🚀 迭代开发 或 🔄 更新Spec）
     * 更新 `spec.md` 头部的版本信息
   
   **更新模式下的执行流程**：
   - 跳过步骤2（创建新功能并初始化文件）
   - 跳过步骤2.d中的版本管理初始化（因为已有版本管理结构）
   - 直接执行步骤3（加载模板）和步骤3.5（结构化设计）
   - 在步骤4执行前，先执行版本管理（创建新版本）
   - 步骤4中更新现有Spec文件（而不是创建新文件）
   - 其他步骤保持不变

2. **创建新功能并初始化文件**（仅在 `$UPDATE_MODE=false` 时执行）：
   
   a. 为短名称找到规格目录中最高的功能编号：
      - 规格目录：检查匹配 `docs/specs/[0-9]+-<short-name>` 的目录
   
   b. 确定下一个可用号码：
      - 从规格目录提取所有号码
      - 找到最高号码 N
      - 为新功能编号使用 N+1
   
   c. 使用计算出的编号和短名称运行脚本 `.specify/scripts/powershell/create-new-feature.ps1 -Json "$ARGUMENTS"`：
      - 传递 `--number N+1` 和 `--short-name "your-short-name"` 以及功能描述
      - Bash 示例：`.specify/scripts/powershell/create-new-feature.ps1 -Json "$ARGUMENTS" --json --number 5 --short-name "user-auth" "Add user authentication"`
      - PowerShell 示例：`.specify/scripts/powershell/create-new-feature.ps1 -Json "$ARGUMENTS" -Json -Number 5 -ShortName "user-auth" "Add user authentication"`
   
   **重要**：
   - 检查规格目录以找到最高编号
   - 仅匹配具有确切短名称模式的目录
   - 如果没有找到此短名称的现有目录，则从编号 1 开始
   - 每个功能只能运行一次此脚本
   - JSON 在终端中作为输出提供 - 始终参考它以获取您正在寻找的实际内容
   - JSON 输出将包含 FEATURE_NAME 和 SPEC_FILE 路径
   - 对于参数中的单引号如"I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或如果可能使用双引号："I'm Groot"）

   d. **立即创建并初始化规格文件**（⚠️ 关键优化：避免过程中断导致文件无法保存）：
      - 加载 `.specify/templates/spec-template.md` 模板
      - 创建初始规格文件，包含：
        * 文档头部（功能编号、创建时间、状态、输入来源、版本信息）
        * **版本管理初始化**（⚠️ 必须执行，仅在创建新功能时）：

           ```markdown
           | B      | [第二个建议的答案] | [这对功能意味着什么] |
           | C      | [第三个建议的答案] | [这对功能意味着什么] |
           | 自定义 | 提供您自己的答案 | [解释如何提供自定义输入] |
           
           **您的选择**：_[等待用户响应]_
           ```

        4. 按顺序编号问题（Q1、Q2、Q3 - 最多 5 个），在等待响应之前一起呈现所有问题
        5. 等待用户回复所有问题的选择（例如，"Q1：A，Q2：自定义 - [详细信息]，Q3：B"）
        6. 通过用用户选择或提供的答案替换每个 [需要澄清] 标记来更新规格
        7. 解决所有澄清后重新运行验证

   d. **更新检查表**：在每次验证迭代后，使用当前通过/失败状态更新检查表文件

7. 报告完成，包括功能名称、规格文件路径、检查表结果以及下一阶段的准备情况（`/澄清需求` 或 `/原型设计` 或 `/制定计划`）。
   
   **保存来源资料**（如果适用）：
   - 如果使用了**探索性对话模式**或**引导式澄清模式**：
     - 在 `FEATURE_DIR/` 创建 `exploration-conversation.md` 记录完整对话过程
     - 在规格文档的"输入"部分引用对话记录
   - 如果是从**禅道需求链接**生成：
     - 在 `FEATURE_DIR/` 创建 `zentao-source.md` 记录禅道来源信息（使用 `sourceInfo.zentaoSourceMarkdown`）
     - 在 `FEATURE_DIR/zentao-screenshots/` 保存页面截图（使用 `sourceInfo.screenshot`，如果有）
   - 如果是从**在线网页/原型**生成：
     - 在 `FEATURE_DIR/` 创建 `web-source.md` 记录网页来源信息（使用 `sourceInfo.webSourceMarkdown`）
     - 在 `FEATURE_DIR/web-source-screenshots/` 保存页面截图（使用 `sourceInfo.screenshot`）
   - 如果是从**HTML原型**生成：
     - 在功能目录下创建 `prototype-source.md` 记录原型来源
   - 如果是从**图片**生成：
     - 在功能目录下创建 `image-source.md` 记录图片来源

**注意：** 脚本在写入之前创建新功能目录并初始化规格文件。

## 一般准则

- 专注于用户需要**什么**以及**为什么**
- 避免如何实施（没有技术栈、API、代码结构）
- 为业务利益相关者而不是开发人员编写
- 不要创建嵌入在规格中的任何检查表
- **新增**：所有技术细节必须转换为业务描述（遵循步骤3.5.b的过滤规则）
- **新增**：功能摘要是必填部分，帮助读者快速理解功能全貌
- **新增**：使用结构化设计方法，先分析再填充细节（步骤3.5）

### 部分要求

- **强制性部分**：必须为每个功能完成
- **可选部分**：仅在与功能相关时包括
- 当部分不适用时，完全删除它（不要留下"N/A"）

### 用于 AI 生成

从用户提示创建此规格时：

1. **需求明确度评估**（⚠️ 重要，仅适用于文本描述输入）：
   - 首先评估需求的明确度（步骤0.5）
   - 如果明确度 < 50分：建议使用探索性对话模式
   - 如果明确度 50-79分：建议使用引导式澄清模式
   - 如果明确度 ≥ 80分：使用标准流程
   - **注意**：对于禅道、网页、HTML、图片等结构化输入，跳过明确度评估，直接使用标准流程

2. **探索性对话模式**（明确度 < 50分时）：
   - **主动提供思路**：每轮对话都要提供行业最佳实践、功能扩展建议、设计思路
   - **渐进式澄清**：从核心到细节，逐步深入
   - **灵活对话**：支持用户随时提问、深入讨论、切换话题
   - **记录对话**：每轮对话后更新 `exploration-conversation.md`
   - **需求总结**：对话结束后，汇总所有已明确的需求点，等待用户确认

3. **引导式澄清模式**（明确度 50-79分时）：
   - **先提供思路**：第1轮提供行业最佳实践、功能扩展建议、设计思路
   - **关键问题**：第2轮一次性提出3-5个关键问题
   - **深入提问**：第3轮根据回答提出1-2个深入问题（可选）
   - **需求确认**：最后1轮汇总需求，等待用户确认

4. **标准流程**（明确度 ≥ 80分，或探索/引导模式完成后）：
   - **进行有根据的猜测**：使用上下文、行业标准和常见模式来填补空白
   - **记录假设**：在假设部分记录合理的默认值
   - **限制澄清**：最多 5 个 [需要澄清] 标记 - 仅用于关键决策：
     - 显着影响功能范围或用户体验
     - 具有不同含义的多种合理解释
     - 缺少任何合理的默认值
   - **优先考虑澄清**：范围 > 安全/隐私 > 用户体验 > 技术细节
   - **像测试人员一样思考**：每个模糊的需求都应该未通过"可测试和明确"检查表项目
   - **需要澄清的常见领域**（仅当不存在合理的默认值时）：
     - 功能范围和边界（包括/排除特定用例）
     - 用户类型和权限（如果可能存在多个冲突的解释）
     - 安全/合规要求（当法律/财务上重要时）
   - **技术细节转换**：
     - 如果输入中包含技术术语，必须先转换为业务描述
     - 参考"结构化设计示例"部分的转换指南
     - 不要将技术实现细节标记为[需要澄清]，而是转换为业务需求
     - 确保所有生成的内容都使用业务语言

5. **对话记录与规格生成**：
   - 探索模式和引导模式的对话记录保存在 `FEATURE_DIR/exploration-conversation.md`
   - 在规格文档的"输入"部分引用对话记录
   - 在规格的"概述"部分说明需求探索过程（如果使用了探索/引导模式）
   - 在"假设"部分记录讨论过的可选方案和扩展功能

**合理默认值的示例**（不要询问这些）：

- 数据保留：领域的行业标准实践
- 性能目标：除非指定，否则标准 Web/移动应用程序期望
- 错误处理：用户友好的消息和适当的回退
- 身份验证方法：Web 应用程序的标准基于会话或 OAuth2
- 集成模式：除非另有说明，否则为 RESTful API

### 成功标准准则

成功标准必须是：

1. **可衡量**：包括具体指标（时间、百分比、计数、速率）
2. **与技术无关**：不提及框架、语言、数据库或工具
3. **以用户为中心**：从用户/业务角度描述结果，而不是系统内部
4. **可验证**：可以在不知道实施细节的情况下进行测试/验证

**好的示例**：

- "用户可以在 3 分钟内完成结账"
- "系统支持 10,000 个并发用户"
- "95% 的搜索在 1 秒内返回结果"
- "任务完成率提高 40%"

**不好的示例**（以实施为重点）：

- "API 响应时间低于 200 毫秒"（太技术性，使用"用户立即看到结果"）
- "数据库可以处理 1000 TPS"（实施细节，使用面向用户的指标）
- "React 组件高效渲染"（特定于框架）
- "Redis 缓存命中率高于 80%"（特定于技术）

## 结构化设计示例

### 技术细节转换示例

| 技术描述 | 业务描述 |
|---------|---------|
| 调用 POST /api/users 接口 | 用户可以通过填写表单创建新用户账户 |
| 使用JWT token进行身份验证 | 系统需要验证用户身份后才能访问功能 |
| 查询users表获取用户列表 | 系统需要显示所有用户的信息列表 |
| 使用Redis缓存提高性能 | 系统需要快速响应用户请求 |
| Vue组件渲染用户列表 | 系统需要在页面上展示用户列表 |

**转换原则**：将技术实现转换为业务能力描述，将API/数据库操作转换为用户操作和系统响应。
