
在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于验证设计实现（原型设计、前端页面设计、前端页面实现）与功能规格文档的一致性，支持四种验证模式：

1. **原型验证**：验证HTML原型页面与规格文档的匹配度
2. **前端设计验证**：验证前端页面设计文档与规格文档的匹配度 ⭐ 新增
3. **前端页面验证**：验证Vue组件实现与规格文档和设计规范的匹配度
4. **规格验证**：验证规格文档本身的完整性和质量

**目标**：
1. 确保设计与规格的一致性
2. 发现需求缺失或理解偏差
3. 验证前端实现是否符合设计规范
4. 提供不匹配时的同步机制
5. 生成验证报告

## 验证模式选择

### 模式1：原型验证（Prototype Validation）

验证HTML原型页面是否准确反映了功能规格文档中定义的需求。

**适用场景**：原型设计完成后、规格有重大变更后、产品评审会议前

**验证内容**：功能需求（FR）匹配、用户故事（US）匹配、数据字段匹配、交互流程匹配

### 模式2：前端设计验证（Frontend Design Validation）⭐ 新增

验证前端页面设计文档是否准确反映了功能规格文档和设计规范。

**适用场景**：前端设计完成后、前端开发前、设计评审会议前

**验证内容**：功能需求匹配、用户故事匹配、数据字段匹配、交互流程匹配、设计规范符合性、组件使用规范性

### 模式3：前端页面验证（Frontend Validation）

验证Vue组件实现是否准确反映了功能规格文档和设计规范。

**适用场景**：前端开发完成后、需要验证实现是否符合需求和设计、代码评审前

**验证内容**：功能需求实现、用户故事流程、数据字段实现、交互行为、设计规范符合性、代码质量

### 模式4：规格验证（Spec Validation）

验证功能规格文档本身的完整性和质量。

**适用场景**：规格文档编写完成后、需要检查规格完整性、准备进入设计阶段前

**验证内容**：功能需求完整性、用户故事完整性、数据模型完整性、验收场景可测试性、术语一致性

## 执行流程

### 1. 模式识别与文件定位

#### 1.1 自动识别验证模式

根据用户输入和项目文件自动识别：
- **原型验证**：用户指定 `--type=prototype` 或检测到 `docs/prototypes/` 目录
- **前端设计验证**：用户指定 `--type=design` 或检测到 `docs/designs/` 目录 ⭐ 新增
- **前端页面验证**：用户指定 `--type=frontend` 或检测到 Vue 文件
- **规格验证**：用户指定 `--type=spec` 或仅指定规格文档路径
- **混合验证**：同时指定多个验证目标

#### 1.2 文件定位

- **规格文档**：用户指定 → 从README.md提取 → 搜索spec.md文件
- **原型文件**（原型验证模式）：用户指定 → 搜索docs/prototypes/最新版本
- **设计文档**（前端设计验证模式）：用户指定 → 搜索docs/designs/最新版本 ⭐ 新增
- **前端文件**（前端验证模式）：用户指定 → 搜索src/views/或src/pages/
- **文件确认**：显示验证模式、文件路径，询问用户确认

### 2. 内容提取与解析

#### 2.1 从规格文档提取验证要素

提取功能需求、用户故事、验收场景、数据字段、页面交互、设计规范要求

#### 2.2 从原型页面提取实现要素（原型验证模式）

提取页面结构、表单字段、数据表格、操作按钮、交互元素

#### 2.3 从前端设计文档提取设计要素（前端设计验证模式）⭐ 新增

提取设计文档中的：
- 页面布局设计
- 组件设计（类型、尺寸、颜色、交互状态）
- 交互设计（操作流程、反馈机制）
- 数据设计（字段、展示方式、筛选排序）
- 样式设计（颜色、间距、字体）
- 设计规范符合性说明

#### 2.4 从前端页面提取实现要素（前端页面验证模式）

提取组件结构、表单字段、数据表格、操作按钮、API调用、交互行为、设计规范符合性

#### 2.5 规格文档质量检查（规格验证模式）

检查完整性、可测试性、一致性、清晰度

### 3. 匹配度验证

对以下维度进行匹配验证：

- **功能需求匹配验证**：检查每个FR在设计/实现中是否有对应功能
- **用户故事匹配验证**：检查每个US在设计/实现中是否支持该流程
- **数据字段匹配验证**：对比规格定义的字段与设计/实现中的字段
- **交互流程匹配验证**：验证设计/实现中的交互是否符合规格定义
- **设计规范符合性验证**（前端设计和前端验证模式）：验证是否符合UI规范、组件使用规范
- **组件使用规范性验证**（前端设计验证模式）：验证组件设计是否符合设计规范要求 ⭐ 新增
- **代码质量验证**（前端验证模式）：验证TypeScript类型、Vue3规范、ESLint等

### 4. 生成验证报告

根据验证模式生成相应的验证报告，包含：
- 基本信息：验证模式、文件路径、验证时间、验证结果
- 匹配度统计：各验证维度的统计表格
- 详细验证结果：各维度的详细匹配表格
- 不匹配项汇总：严重不匹配、一般不匹配、需要人工确认
- 验证结论：根据匹配率给出结论
- 建议操作：根据验证结果给出建议

**报告保存路径**：
- 原型验证：`docs/prototypes/{项目名}/{模块名}/{版本号}/validation-report-{时间戳}.md`
- 前端设计验证：`docs/designs/{项目名}/{模块名}/{版本号}/validation-report-{时间戳}.md` ⭐ 新增
- 前端验证：`{frontend_path}/validation-report-{时间戳}.md`
- 规格验证：`{spec_dir}/spec-validation-report-{时间戳}.md`
- 混合验证：`docs/validation/{项目名}/design-validation-report-{时间戳}.md`

### 5. 不匹配处理（核心流程）

**重要提示**：本命令的核心价值在于**明确展示设计与Spec的差异点，并让用户决定以谁为准**。

如果发现不匹配项，**必须**生成静态HTML页面，让用户一次性查看所有差异并提交处理方式。

#### 5.1 收集所有差异项

对每个差异项，收集以下信息：
- **ID**：差异项唯一标识（diff-001, diff-002...）
- **类型**：功能缺失/字段缺失/实现方式不一致/功能增强/设计规范不符合
- **严重程度**：严重/高/中/低
- **影响范围**：FR编号或功能模块
- **Spec定义**：详细引用规格文档中的原文
- **实现情况**：详细描述设计/实现中的情况
- **可能原因**：分析可能的原因
- **推荐处理方式**：系统根据差异类型推荐的处理方式

#### 5.2 生成差异验证表单

**操作**：
1. 将所有差异项整理为JSON格式，根据验证模式设置`validationMode`：
   - **原型验证模式**：`"validationMode": "prototype"`
   - **前端设计验证模式**：`"validationMode": "design"`
   - **前端页面验证模式**：`"validationMode": "frontend"`
   - **规格验证模式**：`"validationMode": "spec"`

2. 调用脚本生成HTML表单：
   ```bash
   .specify/scripts/bash/generate-difference-validation-form.sh \
     "$VALIDATION_DIR" \
     "$VALIDATION_NAME" \
     "$VALIDATION_DIR/differences.json" \
     "$VALIDATION_DIR/difference-validation-form.html"
   ```

3. 将差异数据保存为JSON文件：`$VALIDATION_DIR/differences.json`
4. 生成HTML表单文件：`$VALIDATION_DIR/difference-validation-form.html`

5. 向用户展示：
   ```
   ════════════════════════════════════════════════════
   检测到 {N} 个差异项
   
   已生成差异验证表单，请按以下步骤操作：
   
   1. 在浏览器中打开以下文件：
      {VALIDATION_DIR}/difference-validation-form.html
   
   2. 查看所有差异项的详细信息（Spec定义、实现情况、可能原因）
   
   3. 为每个差异项选择处理方式：
      - 更新实现（原型/设计文档/前端代码，以Spec为准）
      - 更新Spec（以实现为准）
      - 补充其他意见（填写自定义说明）
      - 跳过（暂不处理）
   
   4. 可以使用批量操作快速设置多个差异项的处理方式
   
   5. 所有差异项处理完成后，点击"提交答案"按钮
   
   6. 系统将下载结果JSON文件，请保存到：
      {VALIDATION_DIR}/difference-validation-results.json
   
   7. 保存完成后，请告诉我"已完成"，系统将继续执行同步操作
   
   ════════════════════════════════════════════════════
   ```

6. **等待用户完成表单填写并保存结果文件**

#### 5.3 读取用户选择并执行同步操作

**操作**：
1. 检查结果文件是否存在：`$VALIDATION_DIR/difference-validation-results.json`
2. 如果不存在，等待用户完成表单填写
3. 读取结果文件，解析用户选择
4. **如果用户选择了"更新Spec"**（⚠️ 必须执行版本管理）：
   - **必须先加载并完全遵循**模块文件：`.cursor/modules/spec-version-management.md`
   - 读取当前版本号（从 `FEATURE_DIR/VERSION` 文件，如果不存在则默认为 v1.0.0）
   - 计算新版本号（Minor版本递增，如：v1.1.0 → v1.2.0）
   - 保存旧版本到 `FEATURE_DIR/versions/{旧版本号}-spec.md`
   - 更新 `FEATURE_DIR/VERSION` 文件为新版本号
   - 准备CHANGELOG记录：
     - 变更类型：🎨 设计验证调整（前端设计验证模式）或根据验证模式确定
     - 变更原因：设计验证阶段发现不匹配项，根据设计/实现调整Spec
     - 变更内容：列出所有更新的差异项
     - 影响范围：说明对开发的影响
     - 验证状态：✅ 已通过设计验证
5. 根据验证模式和用户选择执行同步：
   - **原型验证模式**：
     - 更新原型：解析HTML文件，定位修改位置，生成新代码片段，插入并保存
     - 更新规格：
       * 更新spec.md头部版本信息
       * 解析规格文档，定位修改位置，删除或注释字段定义，补充说明
       * 在版本信息章节添加新版本记录
   - **前端设计验证模式**：
     - 更新设计文档：解析设计文档，定位修改位置，更新设计内容
     - 更新规格：
       * 更新spec.md头部版本信息
       * 解析规格文档，定位修改位置，更新规格内容
       * 在版本信息章节添加新版本记录
   - **前端页面验证模式**：
     - 更新前端代码：解析Vue组件，定位修改位置，更新代码实现
     - 更新规格：
       * 更新spec.md头部版本信息
       * 解析规格文档，定位修改位置，更新规格内容
       * 在版本信息章节添加新版本记录
   - **规格验证模式**：
     - 修复规格文档：根据验证结果修复规格文档中的问题
   - **补充其他意见**：记录用户意见到待处理项，不执行自动同步
   - **跳过**：记录到待处理项

### 6. 生成同步日志

生成同步日志，记录所有同步操作。

### 7. 输出最终报告

输出最终报告，包含验证结果、生成文件、下一步操作建议。

## 命令参数说明

```
/设计验证 [选项]

选项:
  --type=<mode>           指定验证模式（prototype/design/frontend/spec/auto）
  --spec=<path>           指定规格文档路径
  --prototype=<path>      指定原型路径（原型验证模式）
  --design=<path>         指定设计文档路径（前端设计验证模式）⭐ 新增
  --frontend=<path>       指定前端路径（前端页面验证模式）
  --report-only           仅生成验证报告，不执行同步操作
  --auto-fix=<strategy>   自动修复策略（prototype/design/spec/frontend/smart）
  --output=<path>         指定报告输出路径

示例:
  /设计验证                                    # 自动识别模式
  /设计验证 --type=prototype                  # 原型验证
  /设计验证 --type=design                     # 前端设计验证 ⭐ 新增
  /设计验证 --type=frontend                   # 前端页面验证
  /设计验证 --type=spec                       # 规格验证
  /设计验证 --design=docs/designs/user/v1/ --frontend=src/views/user/  # 混合验证
```

## 注意事项

1. **验证时机**：
   - 原型验证：原型设计完成后、前端开发前
   - 前端设计验证：前端设计完成后、前端开发前 ⭐ 新增
   - 前端页面验证：前端开发完成后、测试前
   - 规格验证：规格文档编写完成后、设计前
   - 混合验证：多个设计/实现都完成后，验证整体一致性

2. **验证优先级**：功能完整性 > 设计规范符合性 > 代码质量

3. **前端验证限制**：只能验证静态代码结构，无法验证运行时行为

4. **设计规范识别**：自动识别项目类型，根据项目类型选择对应的设计规范

5. **代码解析能力**：支持Vue3 Composition API、TypeScript、Ant Design Vue和iView组件库
