# 文档解析模块

---
**模块名称**: 文档解析  
**版本**: 1.0  
**创建时间**: 2025-01-27  
**最后更新**: 2025-01-27  
**用途**: 从文档（.doc、.docx、.pdf等）中提取内容并转换为功能描述  
**调用方**: `编写规格.md` 命令  
**依赖模块**: 无（独立模块）  
---

## 📋 模块概述

本模块负责处理各类文档（当前支持.doc、.docx、.pdf格式，可扩展支持其他格式），提取文档中的文本内容、图片、表格等所有信息，分析并生成功能描述文本供主命令继续处理。

**当前支持的文档格式**：
- `.doc` - Microsoft Word 97-2003文档（旧格式）
- `.docx` - Microsoft Word 2007+文档（新格式）
- `.pdf` - PDF文档

**未来可扩展格式**：
- `.md` - Markdown文档
- `.txt` - 纯文本文档
- `.xlsx` - Excel文档（表格数据）
- 其他格式（根据需求扩展）

**核心功能**：
1. 文档格式识别与解析
2. 文本内容全量提取（包括章节结构）
3. 图片提取与分类识别
4. 表格数据提取
5. 内容分析与信息提取
6. 生成功能描述文本

## 🔄 执行流程

### 阶段 0: 输入验证与格式识别

#### 0.1 文件路径验证

**验证规则**:
- ✅ 文件路径必须存在
- ✅ 文件扩展名必须是 `.doc`、`.docx` 或 `.pdf`（不区分大小写）
- ✅ 文件大小不超过100MB（可配置）

**验证失败处理**:
- 文件不存在 → 返回错误：`{ type: 'FILE_NOT_FOUND', message: '文件不存在', path: filePath }`
- 格式不支持 → 返回错误：`{ type: 'UNSUPPORTED_FORMAT', message: '不支持的文档格式，仅支持.doc、.docx、.pdf', path: filePath }`
- 文件过大 → 返回错误：`{ type: 'FILE_TOO_LARGE', message: '文件过大，最大支持100MB', path: filePath, size: fileSize }`

#### 0.2 文档格式识别

**识别方法**:
- 使用文件扩展名识别（主要方法）
- 可选：使用MIME类型验证（提高安全性）

**格式映射**:
- `.doc` → `doc` (Microsoft Word 97-2003)
- `.docx` → `docx` (Microsoft Word 2007+)
- `.pdf` → `pdf` (PDF文档)

**识别结果**:
- 记录文档格式：`{ format: 'docx', path: filePath }`
- 选择对应的解析器

### 阶段 1: 文档解析

#### 1.1 选择解析库

**解析库选择策略**（按优先级）：

**对于.docx格式**:
1. **首选**: `python-docx` - 功能完整，支持文本、表格、图片提取
2. **备用**: `zipfile` - 直接解压docx文件（zip格式），从XML中提取文本

**对于.doc格式**:
1. **首选**: `textract` - 支持多种格式，需要系统依赖
2. **备用1**: `antiword` - 命令行工具，需要系统安装
3. **备用2**: `catdoc` - 命令行工具，需要系统安装
4. **备用3**: 转换为.docx后解析（使用LibreOffice或Microsoft Word）

**对于.pdf格式**:
1. **首选**: `pdfplumber` - 支持文本和表格提取
2. **备用1**: `PyPDF2` - 基础文本提取
3. **备用2**: `pypdf` - PyPDF2的替代版本

**库可用性检测**:
- 尝试导入解析库
- 如果首选库不可用，尝试备用库
- 如果所有库都不可用，返回错误并提示安装

#### 1.2 执行文档解析

**解析内容**:
- 文本内容（段落、标题、列表）
- 文档结构（章节层级）
- 表格数据
- 图片信息（位置、大小、格式）
- 文档元数据（标题、作者、创建时间等）

**解析过程**:
1. 打开文档文件
2. 提取文本内容（保留结构）
3. 提取表格数据
4. 识别图片位置和基本信息
5. 构建文档结构树
6. 提取文档属性（如果可用）

**错误处理**:
- 文件损坏 → 返回错误：`{ type: 'FILE_CORRUPTED', message: '文档文件可能已损坏', path: filePath }`
- 解析失败 → 返回错误：`{ type: 'PARSE_ERROR', message: '文档解析失败', path: filePath, error: errorMessage }`
- 权限不足 → 返回错误：`{ type: 'PERMISSION_DENIED', message: '没有读取文件的权限', path: filePath }`

### 阶段 2: 图片提取与处理

#### 2.1 提取图片文件

**提取策略**:

**对于.docx格式**:
- 从 `word/media/` 目录提取图片文件
- 支持的图片格式：`.png`、`.jpg`、`.jpeg`、`.gif`、`.bmp`、`.emf`、`.wmf`
- 保存图片到临时目录（由主命令指定）
- 处理重名文件（添加序号）

**对于.doc格式**:
- 使用 `textract` 或 `antiword` 提取图片（如果支持）
- 或使用LibreOffice转换后提取
- 图片提取可能不完整（.doc格式限制）

**对于.pdf格式**:
- 使用 `PyPDF2` 或 `pdfplumber` 提取内嵌图片
- 或使用 `pdf2image` 将页面转换为图片（备用方案）
- 提取图片对象并保存

**图片元数据记录**:
```javascript
{
  id: 'img_001',
  originalName: '流程图1.png',
  savedPath: '/tmp/images/flowchart_001.png',
  page: 5,
  chapter: '三、系统功能架构',
  size: 123456,
  format: 'png',
  description: '系统功能架构图'
}
```

#### 2.2 图片分类识别

**分类类型**:
- `flowchart` - 流程图
- `architecture` - 功能架构图
- `sequence` - 时序图
- `ui` - UI截图
- `data_model` - 数据模型图
- `other` - 其他

**识别方法**（按优先级）:
1. **文件名关键词匹配**: 识别文件名中的关键词（流程图、架构图、时序图等）
2. **上下文分析**: 分析图片所在章节和前后文本内容
3. **OCR结果分析**: 如果启用OCR，分析图片中的文字标签
4. **可选**: 使用图像识别模型（如需要更高准确率）

**分类结果**:
- 记录图片类型到元数据
- 如果无法确定类型，标记为 `other`

#### 2.3 OCR识别（可选）

**OCR配置**:
- 默认关闭，可通过参数启用
- 支持中英文识别

**OCR库选择**:
1. **首选**: `PaddleOCR` - 中文识别准确率高，免费
2. **备用**: `pytesseract` - 需要安装Tesseract，轻量级

**OCR处理**:
- 对每张图片执行OCR识别
- 保存OCR结果到图片元数据
- OCR失败不影响整体流程（记录错误但继续处理）

**OCR结果格式**:
```javascript
{
  imageId: 'img_001',
  ocrText: '识别的文字内容...',
  ocrConfidence: 0.85,
  ocrLanguage: 'ch+en'
}
```

### 阶段 3: 内容分析与信息提取

#### 3.1 文档结构分析

**分析内容**:
- 识别标题层级（H1、H2、H3等）
- 构建文档大纲树
- 识别常见的PRD章节（基于关键词匹配）

**常见文档章节关键词**（当前主要针对PRD文档）:
- 概述/Overview/背景
- 用户故事/User Story/Use Case
- 功能需求/Features/功能列表
- 数据模型/Data Model/实体
- API/接口/Interface
- 业务流程/Business Flow
- 异常情况/Exception/Error
- 验收标准/Acceptance Criteria

**结构输出**:
```javascript
{
  title: '文档标题',
  sections: [
    {
      level: 1,
      title: '一、概述',
      content: '...',
      pageStart: 1,
      pageEnd: 3,
      subsections: [...]
    }
  ]
}
```

#### 3.2 关键信息提取

**提取内容**:
- 功能模块名称
- 用户故事
- 业务流程
- 数据模型（字段定义）
- 接口定义
- 验收标准
- 异常情况处理

**提取方法**:
1. **关键词匹配**: 使用正则表达式和关键词匹配提取结构化信息
2. **AI语义分析**: 可选使用AI模型（如GPT）进行语义分析和信息提取
3. **上下文推断**: 结合章节结构和上下文信息提高提取准确性

**提取结果**:
```javascript
{
  modules: ['用户管理', '订单管理', ...],
  userStories: [
    {
      title: '用户注册',
      description: '...',
      acceptanceCriteria: [...]
    }
  ],
  dataModels: [
    {
      name: '用户',
      fields: [
        { name: 'id', type: 'string', required: true, ... }
      ]
    }
  ],
  // ...
}
```

#### 3.3 图片上下文关联

**关联分析**:
- 识别图片所在章节
- 提取图片前后各200字的文本作为上下文
- 分析上下文关键词，推断图片描述
- 建立图片与相关文本的关联关系

**关联结果**:
```javascript
{
  imageId: 'img_001',
  context: {
    chapter: '三、系统功能架构',
    page: 5,
    beforeText: '系统架构如下图所示：',
    afterText: '架构图展示了系统的三层结构...',
    inferredDescription: '系统功能架构图'
  }
}
```

### 阶段 4: 表格数据处理

#### 4.1 表格提取

**提取方法**:

**对于.docx格式**:
- 使用 `python-docx` 提取表格
- 遍历所有表格，提取单元格内容

**对于.pdf格式**:
- 使用 `pdfplumber` 提取表格（优先）
- 或使用OCR识别表格区域（备用）

**对于.doc格式**:
- 表格提取可能不完整（格式限制）
- 尝试使用转换工具后提取

#### 4.2 表格转换

**转换格式**:
- 将表格转换为Markdown表格格式
- 识别表格类型（基于表头关键词）：
  - 数据模型表格（字段名、类型、说明等）
  - 功能列表表格
  - 状态流转表格
  - 其他

**转换结果**:
```markdown
### 数据模型表格

| 字段名 | 类型 | 说明 | 必填 |
|--------|------|------|------|
| id | string | 用户ID | 是 |
| name | string | 用户名 | 是 |
```

- **来源信息**: 文档路径、文档格式、解析时间、文档页数
- **文档摘要**: 文档标题、核心功能概述
- **功能模块列表**: 从文档中提取的功能模块

- **功能需求**: 功能需求列表（标注来源：文本/图片OCR/表格）

- **图片资源**: 图片列表和描述

## 🔧 辅助函数库

### 文档解析函数

- **`detectDocumentFormat(filePath)`**: 检测文档格式
- **`parseDocx(filePath)`**: 解析.docx文档
- **`parseDoc(filePath)`**: 解析.doc文档
- **`parsePdf(filePath)`**: 解析.pdf文档
- **`extractText(document)`**: 提取文本内容
- **`extractTables(document)`**: 提取表格数据
- **`extractDocumentStructure(document)`**: 提取文档结构

### 图片处理函数

- **`extractImages(document, outputDir)`**: 提取图片文件
- **`classifyImage(imageInfo, context)`**: 分类识别图片类型
- **`processOCR(imagePath, options)`**: OCR识别图片文字
- **`saveImage(imageData, outputPath)`**: 保存图片文件

### 内容分析函数

- **`analyzeDocumentStructure(content)`**: 分析文档结构
- **`extractKeyInformation(content)`**: 提取关键信息
- **`associateImageContext(images, content)`**: 关联图片上下文
- **`identifyDocumentSections(structure)`**: 识别文档章节（当前主要针对PRD文档）

### 表格处理函数

- **`extractTablesFromDocx(document)`**: 从docx提取表格
- **`extractTablesFromPdf(document)`**: 从pdf提取表格
- **`convertTableToMarkdown(table)`**: 转换表格为Markdown
- **`identifyTableType(table)`**: 识别表格类型

### 工具函数

- **`validateFilePath(filePath)`**: 验证文件路径
- **`checkLibraryAvailability(libraryName)`**: 检查库可用性
- **`generateImageId(index)`**: 生成图片ID
- **`normalizeImageName(originalName)`**: 规范化图片文件名

## ⚠️ 注意事项

### 通用注意事项

1. **文件路径安全**
   - 验证文件路径，防止路径遍历攻击
   - 使用`pathlib`规范化路径
   - 限制文件大小（默认100MB）

2. **错误处理**
   - 所有解析步骤都要有错误处理
   - 提供清晰的错误提示
   - 部分失败不影响整体流程（如图片提取失败）

3. **性能考虑**
   - 大文档（100+页）使用流式处理
   - 图片处理可以并行（如果支持）
   - OCR处理耗时，默认关闭

4. **格式限制**
   - `.doc`格式支持不完整，优先建议使用`.docx`
   - PDF中的复杂表格可能提取不准确
   - 扫描版PDF需要OCR支持

### 本模块特有注意事项

1. **解析库依赖**
   - 确保必要的解析库已安装
   - 提供降级方案（备用库）
   - 如果所有库都不可用，给出明确的安装提示

2. **图片提取**
   - `.doc`格式的图片提取可能不完整
   - PDF中的图片提取取决于PDF结构
   - 图片文件名冲突时自动添加序号

3. **内容提取准确性**
   - 内容提取基于关键词匹配，可能不够准确
   - 建议结合AI模型提高准确率（可选）
   - 提取结果需要人工验证

4. **OCR识别**
   - OCR识别耗时，默认关闭
   - OCR准确率取决于图片质量
   - OCR失败不影响整体流程

5. **表格处理**
   - 复杂表格可能提取不完整
   - PDF表格提取取决于PDF结构
   - 表格类型识别基于表头关键词

6. **文档结构识别**
   - 文档章节识别基于关键词匹配
