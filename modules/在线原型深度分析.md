---

**用途**: 深度分析在线设计工具（MasterGo、Figma等）的原型文件，确保所有图层、所有功能点、所有UI元素都被完整提取和分析  
**调用方**: `编写规格.md` 命令

## 📋 模块概述

本模块专门用于深度分析在线设计工具（MasterGo、Figma等）的原型文件，确保**所有图层、所有功能点、所有UI元素**都被完整提取和分析，避免遗漏任何设计细节。

## 适用范围

本模块适用于以下设计工具：
- **MasterGo**：国产设计工具，URL包含 `mastergo.com`
- **Figma**：国际设计工具，URL包含 `figma.com` 或 `figma.io`
- **其他设计工具**：Axure、Sketch等（需要根据实际情况调整）

## 🔄 执行流程

**引用**: [网页解析基础模块.md#阶段1-URL验证](./网页解析基础模块.md#阶段-1-url-验证) 和 [网页解析基础模块.md#阶段2-页面访问](./网页解析基础模块.md#阶段-2-页面访问)

**本模块特定处理**:
- **URL检测**: 识别URL中包含 `mastergo.com`、`figma.com` 或 `figma.io` 的链接
- **页面类型识别**: 检测页面标题是否包含"设计文件"、"MasterGo"、"Figma"等关键词
- **等待时间**: 设计工具页面需要5-10秒等待图层完全加载
- **登录处理**: 采用策略C（通常不需要登录，设计工具使用公开分享链接）

**本模块特定处理**:
- 必须提取HTML结构用于分析DOM结构
- 需要等待设计工具完全渲染（可能需要额外等待2-3秒）

## 工具差异说明

### MasterGo vs Figma 相同点

1. **图层结构**：都采用树形图层结构，支持图层分组和嵌套
2. **UI元素**：都包含统计卡片、表格、图表、按钮、表单等常见UI元素
3. **设计规范**：都需要提取颜色、间距、字体、圆角、阴影等设计规范
4. **交互状态**：都支持hover、active、disabled等交互状态
5. **组件系统**：都支持组件化和复用

### MasterGo vs Figma 不同点

1. **URL结构**：
   - MasterGo: `https://mastergo.com/file/{fileId}?...`
   - Figma: `https://www.figma.com/file/{fileId}/{fileName}?...` 或 `https://www.figma.com/design/{fileId}/...`

2. **DOM结构**：
   - MasterGo: 使用特定的DOM结构和类名
   - Figma: 使用React渲染，DOM结构可能不同

3. **图层命名**：
   - MasterGo: 中文命名较多
   - Figma: 英文命名较多

4. **分享方式**：
   - MasterGo: 支持查看模式和编辑模式
   - Figma: 支持查看、评论、编辑等多种权限

5. **组件库**：
   - MasterGo: 可能有特定的组件库结构
   - Figma: 使用Frames和Components组织

### 分析策略

**核心分析流程相同**：两个工具的分析流程（5个阶段）完全相同，因为：
- UI元素类型相同（统计卡片、表格、图表等）
- 功能逻辑推断方法相同
- 设计规范提取方法相同
- 完整性验证标准相同

**实现细节需要适配**：
- 图层选择器可能需要调整
- DOM遍历方式可能需要调整
- 但分析逻辑和检查清单可以完全复用

## 问题分析

### 为什么会出现未分析到位的情况？

1. **图层可见性问题**
   - MasterGo设计文件可能包含大量图层，部分图层可能被折叠或隐藏
   - 某些图层需要滚动或缩放才能看到完整内容
   - 交互状态（hover、active等）可能在不同图层中

2. **分析深度不足**
   - 只分析了主要图层，忽略了子图层和嵌套结构
   - 没有深入分析每个UI组件的所有状态和属性
   - 遗漏了统计卡片、标签页、视图切换等细节功能

3. **功能推断不完整**
   - 只提取了明显的功能，忽略了隐含的交互逻辑
   - 没有分析数据流向和状态管理
   - 遗漏了筛选、排序、分页等辅助功能

4. **UI元素提取不全面**
   - 只提取了主要组件，忽略了装饰性元素和状态指示器
   - 没有提取颜色、间距、字体等设计规范
   - 遗漏了响应式布局和适配规则

## 深度分析流程（设计文件特有）

### 阶段1：完整图层扫描（⚠️ 必须执行）

**目标**：确保所有图层都被识别和分析

1. **获取完整图层树**
   ```javascript
   // 使用浏览器工具获取完整DOM结构
   // 包括所有折叠的图层和隐藏的图层
   ```

2. **图层分类**
   - **页面图层**：完整的页面设计（如"设备监控"、"成本分析"）
   - **组件图层**：可复用的UI组件（如统计卡片、表格、图表）
   - **状态图层**：不同交互状态（hover、active、disabled）
   - **装饰图层**：图标、分割线、背景等

3. **图层遍历策略**
   - 从根节点开始，递归遍历所有子节点
   - 记录每个图层的层级关系
   - 识别图层分组和命名规则

### 阶段2：UI元素深度提取（⚠️ 必须执行）

**目标**：提取每个图层中的所有UI元素和属性

#### 2.1 统计卡片分析
- [ ] 卡片数量
- [ ] 每个卡片的标题
- [ ] 每个卡片的值（数字、百分比、单位）
- [ ] 每个卡片的变化趋势（上升/下降、变化值）
- [ ] 每个卡片的图标或装饰元素
- [ ] 卡片的布局方式（网格、单行等）

#### 2.2 筛选器分析
- [ ] 筛选字段列表（设备名称、设备类型、状态等）
- [ ] 每个筛选字段的输入类型（输入框、下拉框、日期选择器等）
- [ ] 筛选字段的默认值或占位符
- [ ] 筛选按钮（查询、重置、刷新等）
- [ ] 筛选器的布局方式（行内、块级等）

#### 2.3 标签页分析
- [ ] 标签页数量
- [ ] 每个标签页的名称和计数（如"电能表(10)"）
- [ ] 标签页的激活状态
- [ ] 标签页切换的交互逻辑

#### 2.4 视图切换分析
- [ ] 视图类型（网格视图、列表视图）
- [ ] 切换按钮的位置和样式
- [ ] 当前激活的视图类型

#### 2.5 表格分析
- [ ] 表格列定义（列名、宽度、对齐方式）
- [ ] 表格列的类型（文本、数字、日期、状态、操作）
- [ ] 状态列的显示方式（标签、徽章、图标）
- [ ] 操作列的按钮（查看详情、编辑、删除等）
- [ ] 表格的分页配置（页码、每页条数、快速跳转）
- [ ] 表格的排序功能
- [ ] 表格的筛选功能

#### 2.6 图表分析
- [ ] 图表类型（柱状图、折线图、饼图、环形图等）
- [ ] 图表的标题和说明
- [ ] 图表的数据维度（X轴、Y轴、系列）
- [ ] 图表的颜色配置
- [ ] 图表的交互功能（tooltip、缩放、图例等）

#### 2.7 按钮分析
- [ ] 按钮类型（主要按钮、次要按钮、链接按钮）
- [ ] 按钮的文本和图标
- [ ] 按钮的位置（顶部、底部、操作列等）
- [ ] 按钮的状态（正常、禁用、加载中）

#### 2.8 表单分析
- [ ] 表单字段列表
- [ ] 每个字段的类型、标签、占位符
- [ ] 字段的验证规则（必填、格式、长度等）
- [ ] 表单的提交和重置逻辑

### 阶段3：功能逻辑推断（⚠️ 必须执行）

**目标**：从UI元素推断完整的功能逻辑

1. **数据流向分析**
   - 筛选器 → 数据查询 → 表格/图表更新
   - 标签页切换 → 数据重新加载
   - 视图切换 → 数据展示方式改变

2. **交互流程分析**
   - 用户操作 → 系统响应 → UI更新
   - 状态变化 → UI反馈
   - 错误处理 → 错误提示

3. **业务规则推断**
   - 从UI约束推断业务规则（如必填字段、数据格式）
   - 从状态显示推断业务状态（如在线/离线/故障）
   - 从操作按钮推断业务操作（如查看、编辑、删除）

### 阶段4：设计规范提取（⚠️ 必须执行）

**目标**：提取完整的设计规范，确保UI实现与设计一致

1. **颜色系统**
   - 主色调（蓝色、绿色、橙色等）
   - 文本颜色（主文本、次要文本、辅助文本）
   - 背景颜色（页面背景、卡片背景）
   - 状态颜色（成功、警告、错误、信息）

2. **间距系统**
   - 卡片间距
   - 组件间距
   - 内边距和外边距

3. **字体系统**
   - 字体大小（标题、正文、辅助文本）
   - 字体粗细（正常、中等、粗体）
   - 行高和字间距

4. **圆角和阴影**
   - 卡片圆角
   - 按钮圆角
   - 阴影效果

### 阶段5：完整性验证（⚠️ 必须执行）

**检查清单**：

- [ ] 所有页面图层都已分析
- [ ] 所有统计卡片都已提取（数量、标题、值、趋势）
- [ ] 所有筛选字段都已提取（字段名、类型、选项）
- [ ] 所有标签页都已提取（名称、计数、激活状态）
- [ ] 所有视图切换功能都已识别
- [ ] 所有表格列都已提取（列名、类型、宽度、操作）
- [ ] 所有图表都已提取（类型、标题、数据维度）
- [ ] 所有按钮都已提取（文本、类型、位置）
- [ ] 所有状态显示都已提取（在线、离线、故障等）
- [ ] 所有交互逻辑都已推断（筛选、排序、分页、切换）
- [ ] 所有设计规范都已提取（颜色、间距、字体、圆角、阴影）

## 使用指南

### 在编写规格命令中集成

1. **检测设计文件类型**
   - **MasterGo**：识别URL中包含 `mastergo.com` 的链接，或页面标题包含"设计文件"或"MasterGo"
   - **Figma**：识别URL中包含 `figma.com` 或 `figma.io` 的链接，或页面标题包含"Figma"
   - **其他工具**：根据实际情况识别

2. **执行深度分析流程**
   - 先执行通用流程（URL验证、页面访问、登录处理）
   - 然后执行设计文件特有的5个阶段（图层扫描、UI元素提取、功能推断、设计规范提取、完整性验证）
   - 使用检查清单确保完整性（所有工具通用）
   - **工具特定适配**：
     - MasterGo：使用MasterGo特定的DOM选择器
     - Figma：使用Figma特定的DOM选择器
     - 但分析逻辑和检查清单完全相同

3. **生成详细的分析报告**
   - 保存到 `web-source.md`
   - 包含完整的图层列表和UI元素清单
   - 标注设计工具类型（MasterGo/Figma/其他）

4. **基于分析结果生成规格**
   - 确保所有功能点都被包含在规格中
   - 确保所有UI元素都被描述
   - 确保所有交互逻辑都被定义

### 工具特定实现细节

#### MasterGo特定处理
- DOM选择器：可能需要使用特定的类名或ID
- 图层遍历：可能需要处理MasterGo特定的图层结构
- 截图方式：可能需要特殊处理

#### Figma特定处理
- DOM选择器：Figma使用React，可能需要等待React渲染完成
- 图层遍历：Figma的图层结构可能使用不同的DOM结构
- 权限处理：Figma可能有查看、评论、编辑等不同权限模式
- 组件识别：Figma使用Frames和Components，需要特殊识别

#### 通用处理逻辑
- UI元素提取逻辑：完全相同（统计卡片、表格、图表等）
- 功能逻辑推断：完全相同
- 设计规范提取：完全相同
- 完整性验证：完全相同

## ⚠️ 注意事项

### 通用注意事项

**引用**: [网页解析基础模块.md#注意事项](./网页解析基础模块.md#注意事项)

### 设计文件特有注意事项

1. **不要假设**：对于不确定的元素，必须通过浏览器工具确认
2. **不要遗漏**：即使是装饰性元素，也可能包含重要的设计信息
3. **不要简化**：详细记录每个元素的属性，不要合并或简化
4. **多次验证**：完成分析后，再次检查是否还有遗漏的元素
5. **图层可见性**：注意处理折叠和隐藏的图层
6. **等待渲染**：设计工具页面需要额外等待时间确保图层完全加载
