# 禅道需求解析模块

---
**模块名称**: 禅道需求解析  
**版本**: 2.0  
**创建时间**: 2025-01-XX  
**最后更新**: 2025-01-XX  
**用途**: 从禅道Story链接中提取需求内容并转换为功能描述  
**调用方**: `编写规格.md` 命令  
**依赖模块**: [网页解析基础模块.md](./网页解析基础模块.md)  
---

## 📋 模块概述

本模块负责处理禅道（Zentao）Story需求链接，使用 Browser 工具访问页面，提取需求描述、图片和附件内容，分析并生成功能描述文本供主命令继续处理。

---

## 🔄 执行流程

### 阶段 1: URL 验证

**引用**: [网页解析基础模块.md#阶段1-URL验证](./网页解析基础模块.md#阶段-1-url-验证)

**本模块特定处理**:
- **URL检测**: URL中必须包含 `zentao` 和 `story`（不区分大小写）
- **验证失败提示**: "您提供的链接不是禅道Story需求链接，请重新输入包含'zentao'和'story'的链接"
- **验证通过**: 记录URL地址，准备访问页面

---

### 阶段 2: 页面访问与登录处理

**引用**: [网页解析基础模块.md#阶段2-页面访问](./网页解析基础模块.md#阶段-2-页面访问) 和 [网页解析基础模块.md#阶段3-登录处理](./网页解析基础模块.md#阶段-3-登录处理)

**本模块特定处理**:
- **等待时间**: 3-5秒（禅道页面通常加载较快）
- **登录处理**: 采用策略A（等待用户手动登录并确认）
- **登录提示模板**: 使用标准模板（见网页解析基础模块）
- **登录验证**: 等待用户确认后，重新获取快照验证登录结果

---

### 阶段 2.5: 获取页面内容

**引用**: [网页解析基础模块.md#阶段4-页面内容获取](./网页解析基础模块.md#阶段-4-页面内容获取)

**本模块特定处理**:
- 必须获取页面快照用于提取需求内容
- 可选获取截图（如果需求描述中包含图片）

### 阶段 3: 需求内容提取（禅道特有）

#### 3.1 提取需求描述

**操作**:
- 使用 `browser.snapshot()` 获取页面可访问性快照
- 从快照中提取需求描述文本内容
- 识别需求描述的主要区域（通常包含：标题、描述、验收标准等）
- 提取以下信息：
  - Story标题
  - 需求描述
  - 验收标准
  - 优先级
  - 状态
  - 其他元数据

#### 3.2 提取并处理图片

**识别需求描述中的图片**:
- 从页面快照中识别所有图片元素（img标签、背景图片等）
- 检查图片是否在需求描述区域内
- 记录图片的URL或路径信息

**如果存在图片**:
1. **下载或截图保存图片**:
   - 对于可访问的图片URL，尝试下载图片
   - 或使用 `browser.screenshot()` 截取图片区域
   - 保存图片到临时目录（由主命令处理）

2. **使用OCR识别图片内容**:
   - 对每张图片执行OCR识别（使用 `image_to_markdown.py` 或类似工具）
   - 收集所有图片的识别结果
   - 合并图片识别结果

#### 3.3 提取并处理附件

**识别附件**:
- 从页面快照中识别附件链接（通常包含"附件"、"下载"、"Download"等关键词）
- 检查附件是否与需求相关
- 记录附件的URL和文件名

**如果存在附件**:
1. **下载附件**:
   - 对于可访问的附件URL，下载附件文件
   - 保存附件到临时目录（由主命令处理）

2. **提取附件内容**:
   - 对于PDF格式的附件，使用 `pdf_to_markdown.py` 提取内容
   - 对于其他格式的附件，记录附件信息

3. **合并附件内容**:
   - 将所有附件的提取内容合并

#### 3.4 合并所有内容

- 将需求描述、图片识别结果、附件内容合并
- 生成统一的内容摘要

---

### 阶段 4: 分析需求信息

根据合并的内容，分析并提取：
- 功能模块名称（从需求描述中提取）
- 用户故事和验收场景（从需求描述中提取）
- 功能需求点（从需求描述、图片OCR、附件内容中提取）
- 交互流程和规则（从需求描述中整理）
- 数据字段和验证规则（从需求描述中提取）
- 状态和操作说明（从需求描述中整理）

**重要提示**：
- 只关注功能需求相关的分析和提取，不考虑翻页、日志记录、数据同步等通用或非本次需求指定的系统能力
- 基于提取的内容进行合理推断和补充，但需在"未明确需求"中标注需要进一步确认的内容
- 对于描述模糊的部分，记录为待澄清项

---

### 阶段 5: 生成功能描述文本

将所有提取和推断的信息整合为结构化的功能描述文本。

**包含内容**:
- 来源信息（URL、Story ID、标题、优先级、状态）
- 核心功能列表
- 用户故事
- 功能需求（标注来源：需求描述/图片OCR/附件内容）
- 数据模型（如果涉及）
- 待澄清项（如果有）

---

### 阶段 6: 保存原始资料

在返回功能描述之前，准备好原始资料元数据，供主命令保存。

**包含内容**:
- 来源类型、原始URL、访问时间、Story标题、Story ID
- 截图路径（如果有）
- 提取的需求内容统计
- 附件信息（如果有）

## 🔧 辅助函数库

- **validateZentaoUrl()**: 验证URL是否为有效的禅道Story链接
- **extractStoryContent()**: 从页面快照中提取Story内容
- **extractImages()**: 识别并提取需求描述中的图片
- **extractAttachments()**: 识别并提取附件
- **mergeContent()**: 合并需求描述、图片OCR、附件内容
- **analyzeRequirements()**: 分析需求内容并提取功能需求

## 📝 使用示例

### 示例: 单Story提取

**输入**: `https://zentao.example.com/story/view/123`

**处理流程**: 
1. 验证URL → 包含'zentao'和'story'，验证通过
2. 访问页面 → 检测到需要登录 → 提示用户登录 → 等待确认
3. 提取需求描述 → 提取图片 → 提取附件
4. 合并内容 → 分析需求 → 生成功能描述

**输出**: 功能描述文本 + 原始资料

## ⚠️ 注意事项

### 通用注意事项

**引用**: [网页解析基础模块.md#注意事项](./网页解析基础模块.md#注意事项)

### 本模块特有注意事项

1. **URL验证**
   - 必须验证URL包含 `zentao` 和 `story`
   - 如果验证失败，提示用户重新输入
   - 不通过验证不继续后续流程

2. **登录处理**
   - 采用策略A：等待用户手动登录并确认
   - **请用户帮忙登录**（不自动输入用户名密码）
   - 等待用户确认登录完成后再继续
   - 验证登录是否成功

3. **内容提取**
   - 确保页面完全加载后再提取内容
   - 处理动态加载的内容（可能需要等待）
   - 提取需求描述、图片、附件三种内容

4. **图片处理**
   - 只处理需求描述区域内的图片
   - 使用OCR工具进行图片识别
   - 图片识别结果作为需求分析的输入源

5. **附件处理**
   - 优先处理PDF格式的附件
   - 使用PDF提取工具提取内容
   - 附件内容作为需求分析的输入源

6. **范围约束**
   - 不考虑超时、网络异常、大数据量等边界情况
   - 不考虑日志、同步等非本次需求指定的功能
   - 需仅围绕需求本身整理成功能描述
