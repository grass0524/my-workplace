## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于将大型Spec文件按模块拆分为多个文件，以降低单次对话的上下文长度，同时保持模块关联性和引用关系的完整性。

**支持的Spec类型**：
- ✅ **系统级Spec**：包含多个功能模块的Spec（如`### {模块名称}模块`）
- ✅ **单功能Spec**：包含功能摘要、用户场景、需求等的单个功能Spec（基于新Spec模板）

**核心特性**：
- ✅ **保留原始文件**：原始spec.md文件保持不变，用于后续更新
- ✅ **按模块拆分**：系统级Spec按模块拆分，保持模块内功能的关联性，每个模块一个文件
- ✅ **按用户故事拆分**：单功能Spec按用户故事拆分，便于独立开发和测试
- ✅ **功能摘要保留**：自动识别并保留功能摘要部分在索引文件中，作为快速导航入口
- ✅ **通用部分拆分**：将通用部分（交互流程、业务规则、UI/UX规范）拆分到common/目录
- ✅ **自动识别引用关系**：自动识别模块间的字段引用和数据关联
- ✅ **自动维护引用**：在每个模块文件中自动添加引用标记和链接
- ✅ **生成精简索引文件**：生成spec-index.md作为主索引文件（精简版，约200-350行），只包含索引和导航，不包含通用部分详细内容

## 执行流程

### 阶段0：输入验证与文件检查

#### 0.1 验证输入

**输入格式**：
- 如果指定了Spec文件路径：使用指定路径
- 如果未指定：在当前目录或`docs/specs/`目录下查找`spec.md`文件
- 如果找到多个spec.md文件：列出所有文件让用户选择

**验证规则**：
- ✅ Spec文件必须存在
- ✅ Spec文件必须是Markdown格式
- ✅ Spec文件必须包含以下之一：
  - 功能模块划分（通过"### {模块名称}模块"等标题识别）- 系统级Spec
  - 功能摘要部分（通过"## 功能摘要"标题识别）- 单功能Spec

#### 0.2 检查是否已拆分

**检查项**：
- 检查是否存在`modules/`目录（系统级Spec）或`user-stories/`目录（单功能Spec）
- 检查是否存在`spec-index.md`文件

**如果已拆分**：
- 询问用户是否重新拆分（会覆盖现有拆分文件）
- 如果选择重新拆分，先备份现有拆分文件：
  - 系统级Spec：备份到`modules-backup-{timestamp}/`、`common-backup-{timestamp}/`和`spec-index-backup-{timestamp}.md`
  - 单功能Spec：备份到`user-stories-backup-{timestamp}/`、`requirements-backup-{timestamp}.md`（如果存在）和`spec-index-backup-{timestamp}.md`

#### 0.3 确认原始文件并判断Spec类型

**操作**：
1. 确认原始`spec.md`文件存在且可读
2. 记录原始文件路径，用于后续更新
3. **重要**：原始spec.md文件不会被修改或重命名，保持原样
4. **判断Spec类型**（快速扫描文件内容）：
   - 如果包含`### {模块名称}模块`格式的标题 → **系统级Spec**
   - 如果包含`## 功能摘要`标题且不包含模块标题 → **单功能Spec**
   - 如果同时包含功能摘要和模块标题 → **系统级Spec**（优先按系统级处理）
   - 记录Spec类型，用于后续拆分策略选择

### 阶段1：解析Spec文件结构

#### 1.1 读取并解析Spec文件

**操作**：
1. 读取完整的spec.md文件内容
2. 识别文档结构（根据阶段0.3判断的Spec类型）：

   **通用结构（两种类型都可能有）**：
   - **文档头部**：功能编号、创建时间、状态、输入（单功能Spec）或文档版本、生成日期等（系统级Spec）
   - **功能摘要**：`## 功能摘要 *（必填）*`或`## 功能摘要`（⭐ 新增识别）
     - 核心功能概览
     - 关键业务流程
     - 关键数据实体（快速参考）
     - 关键交互点（快速参考）
   - **目录**（如果存在）

   **系统级Spec结构**：
   - 系统概述
   - 功能架构
   - 功能规格说明（按模块）
   - 页面规格说明
   - 交互流程
   - 业务规则
   - UI/UX规范
   - 附录

   **单功能Spec结构**：
   - 用户场景和测试（`## 用户场景和测试 *（必填）*`）
   - 需求（`## 需求 *（必填）*`）
   - 成功标准（`## 成功标准 *（必填）*`）
   - 可选部分（概述、澄清、假设、依赖项）

3. **记录关键部分位置**：
   - 功能摘要的起始行和结束行（如果存在）
   - 文档头部的起始行和结束行
   - 各主要章节的起始行和结束行

#### 1.2 识别拆分单元边界

**通用操作**：
- 识别拆分单元（模块或用户故事）的标题和内容
- 记录每个拆分单元的起始行和结束行
- 记录拆分单元的元数据（模块名称/用户故事标题、优先级等）

**系统级Spec - 识别模块边界**：
- 查找所有`### {模块名称}模块`格式的模块标题
- 识别每个模块下的功能（`#### F-XXX: 功能名称`或`#### T-XXX: 功能名称`）
- 记录每个模块包含的功能列表
- **识别规则**：
  - 模块标题格式：`### {模块名称}模块`（如"### 系统管理模块"）
  - 功能标题格式：`#### F-XXX: {功能名称}`或`#### T-XXX: {功能名称}`
  - 模块结束：下一个模块标题或通用部分标题（如"## 页面规格说明"、"## 交互流程"等）

**单功能Spec - 识别用户故事边界**：
- 查找所有`### 用户故事 X - {标题}（优先级：P{X}）`格式的用户故事标题
- 识别每个用户故事下的内容（描述、为什么是这个优先级、独立测试、验收场景等）
- 记录每个用户故事的优先级和标题
- 识别边缘情况部分（`### 边缘情况`）
- **识别规则**：
  - 用户故事标题格式：`### 用户故事 {数字} - {标题}（优先级：P{数字}）`
  - 用户故事结束：下一个用户故事标题或"### 边缘情况"或"## 需求"等主要章节标题

#### 1.3 识别通用部分和其他章节

**识别策略**：
- **通用部分**（系统级Spec常见）：
  - `## 页面规格说明`
  - `## 交互流程`
  - `## 业务规则`
  - `## UI/UX规范`
- **单功能Spec的其他章节**（⭐ 新增）：
  - `## 需求 *（必填）*`或`## 需求`
  - `## 成功标准 *（必填）*`或`## 成功标准`
  - `## 概述`（可选）
  - `## 澄清`（可选）
  - `## 假设`（可选）
  - `## 依赖项`（可选）
- 记录每个部分的起始行和结束行

### 阶段2：识别引用关系（仅系统级Spec）

> **注意**：此阶段仅适用于系统级Spec，单功能Spec跳过此阶段。

#### 2.1 字段引用识别

**识别策略**：
1. **表单字段分析**：查找`**表单字段**:`部分，识别下拉选择、树形选择等字段类型，根据字段名称推断引用关系
   - "归属部门"、"所属部门"、"部门" → 引用部门管理功能
   - "岗位"、"岗位选择" → 引用岗位管理功能
   - "角色"、"角色选择" → 引用角色管理功能
   - "权限"、"菜单权限"、"角色权限" → 引用菜单管理功能
2. **筛选条件分析**：查找`**筛选条件**:`部分，识别下拉选择类型，推断引用关系
3. **数据模型分析**：查找`#### {表名}`部分，识别外键字段（如"部门ID"、"角色ID"、"菜单ID"、"岗位ID"），推断引用关系

#### 2.2 业务规则引用识别

**识别策略**：
- 查找业务规则中提到的其他模块或功能
- 识别规则中的依赖关系（如"用户必须属于某个部门"、"角色需要配置菜单权限"）

#### 2.3 信息展示引用识别

**识别策略**：
- 查找"查看"、"显示"等关键词
- 识别信息展示中引用的其他模块（如"所属部门"、"所属角色"）

#### 2.4 建立引用关系映射

**映射数据结构**：源模块名称、源功能ID和名称、目标模块名称、目标功能ID和名称、引用类型（字段引用/数据模型引用/业务规则引用/信息展示引用）、引用字段名称、引用说明

### 阶段3：创建文件结构

#### 3.1 创建目录结构

**通用操作**：
1. 确认原始`spec.md`文件所在目录
2. 如果目录已存在，先备份现有内容到`{目录名}-backup-{timestamp}/`
3. 创建必要的目录和文件

**根据Spec类型创建目录**：

| Spec类型 | 创建的目录/文件 | 说明 |
|---------|---------------|------|
| **系统级Spec** | `modules/` | 功能模块目录 |
| | `common/` | 通用部分目录（交互流程、业务规则、UI/UX规范） |
| **单功能Spec** | `user-stories/` | 用户故事目录 |
| | `requirements.md` | 需求部分（仅当内容>200行时创建） |
| | `success-criteria.md` | 成功标准（仅当内容>100行时创建） |

**目录结构示例**：
```
docs/specs/{编号}-{名称}/
├── spec.md                   # 原始Spec文件（保持不变）
├── spec-index.md             # 索引文件
├── [modules/ 或 user-stories/] # 拆分单元目录
│   └── [拆分单元文件].md
└── [common/]                 # 仅系统级Spec：通用部分目录
    └── [通用文件].md
```

### 阶段4：生成拆分单元文件

#### 4.1 提取拆分单元内容

**通用操作**：
- 对每个识别到的拆分单元（模块或用户故事）：
  1. 提取拆分单元的标题和元数据
  2. 提取拆分单元下的所有内容
  3. 保留完整的格式和结构

**系统级Spec - 提取模块内容**：
- 提取模块标题和描述（如果有）
- 提取模块下的所有功能规格
- 保留功能的完整内容（功能描述、页面URL、功能类型、核心操作、表单字段、筛选条件、业务规则、交互流程、截图等）

**单功能Spec - 提取用户故事内容**：
- 提取用户故事标题、优先级
- 提取用户故事描述、"为什么是这个优先级"、"独立测试"部分
- 提取所有验收场景（假设-当-那么格式）
- 提取边缘情况（如果有，单独生成edge-cases.md）

#### 4.2 添加引用关系标记（仅系统级Spec）

**操作**：
- 对每个功能，根据引用关系映射：
  1. 在功能描述后添加"关联模块"部分（如果有引用），格式：`📎 [功能名称](#f-xxx-功能名称) - 用于"字段名"字段（字段类型）`
  2. 在相关字段后添加引用链接注释，格式：`→ 数据来源：[功能名称](#f-xxx-功能名称)`
  3. 生成引用关系说明，包含引用类型和引用字段名称

#### 4.3 生成拆分单元文件

**系统级Spec - 生成模块文件**：
- 文件标题：`# {模块名称}模块`
- 模块概述：模块描述（如果有）
- 功能列表：所有功能的锚点链接列表
- 功能规格说明：所有功能的详细内容（包含引用关系标记）
- 模块内引用关系：模块内功能间的引用关系图（如果有）

**单功能Spec - 生成用户故事文件**：
- 文件标题：`# 用户故事 {数字} - {标题}（优先级：P{数字}）`
- 用户故事描述：完整描述
- 为什么是这个优先级：优先级说明
- 独立测试：独立测试说明
- 验收场景：所有验收场景（假设-当-那么格式）
- 页脚：添加说明文字，指向原始spec.md和索引文件

---

### 阶段5：生成通用部分文件

#### 5.1 提取通用部分内容

**操作流程**：
- 对每个通用部分（交互流程、业务规则、UI/UX规范）：
  1. 根据阶段1.3识别的起始行和结束行提取完整内容
  2. 保留原有的格式和结构（标题层级、列表、表格、代码块等）
  3. 如果内容中引用了具体功能，保持引用（但可能需要更新为相对路径）
  4. 提取内容包括：
     - 通用部分的二级标题（## 交互流程）
     - 所有子标题和内容
     - 直到下一个通用部分标题或模块引用关系部分为止

#### 5.2 生成通用文件

**文件列表**：
- `common/interaction-flows.md` - 交互流程
- `common/business-rules.md` - 业务规则
- `common/ui-ux-specs.md` - UI/UX规范

**文件结构**：
- 文件标题：通用部分名称（如"# 交互流程"）
- 内容：原始通用部分内容，完整保留原有格式和结构
- 页脚：添加说明文字，指向原始spec.md和索引文件

---

### 阶段6：生成主文件（索引文件）

#### 6.1 统计文件信息

**通用操作**：
1. 统计原始文件行数（spec.md）
2. 统计索引文件预估行数
3. 统计所有拆分单元文件的行数
4. 记录所有文件的路径和描述

**系统级Spec统计数据**：
- 原始文件行数
- 索引文件预估行数（约300-350行）
- 每个模块文件的行数、功能数量、功能ID列表（F-XXX格式）
- 每个通用文件的行数

**单功能Spec统计数据**：
- 原始文件行数
- 索引文件预估行数（约200-300行，包含功能摘要）
- 每个用户故事文件的行数、优先级
- 需求/成功标准文件行数（如果拆分）

#### 6.2 生成文件清单部分

**通用生成内容**：
1. **文件结构图**：使用树形结构展示目录结构
2. **所有Spec文件列表表格**：包含列（文件路径、文件类型、行数、包含内容、用途）
3. **快速导航部分**：按不同维度生成导航链接

**系统级Spec特有**：
- **功能模块映射表**：模块名称、模块文件、功能数量、功能ID列表
- 导航维度：按开发任务、按功能ID、按通用规范

**单功能Spec特有**：
- **用户故事映射表**：用户故事标题、优先级、文件路径
- 导航维度：按优先级、按用户故事

#### 6.3 生成主文件内容

**索引文件结构**（根据Spec类型生成不同的结构）：

**系统级Spec**（保存为spec-index.md，精简版，约300-350行）：

1. **文档头部**：标题、说明文字、文档信息
2. **Spec文件清单**：文件结构图、所有文件列表表格、功能模块映射表、快速导航
3. **目录**：6个主要章节的锚点链接
4. **系统概述**：原始系统概述内容（如果包含功能摘要，也保留在此）
5. **功能架构**：原始功能架构内容
6. **功能规格说明**：各模块的索引链接（不包含详细内容，只包含模块名称、功能数量、功能列表）
7. **页面规格说明**：原始页面规格说明内容
8. **通用规范**：通用部分的索引链接（不包含详细内容，只包含链接到common/目录的文件）
9. **模块引用关系**：引用关系图（Mermaid）和模块依赖表
10. **附录**：原始附录内容

**单功能Spec**（保存为spec-index.md，精简版，约200-300行）（⭐ 新增）：

1. **文档头部**：功能编号、创建时间、状态、输入
2. **功能摘要**（⭐ 关键优化）：完整保留功能摘要内容
   - 核心功能概览
   - 关键业务流程
   - 关键数据实体（快速参考）
   - 关键交互点（快速参考）
   - **注意**：功能摘要必须保留在索引文件中，作为快速导航入口，帮助读者5分钟内理解功能全貌
3. **Spec文件清单**：文件结构图、所有文件列表表格、快速导航
4. **用户故事索引**：链接到user-stories/目录的文件，包含优先级和简要描述
5. **需求索引**：链接到requirements.md或显示简要内容（如果内容较短，直接显示）
6. **成功标准索引**：链接到success-criteria.md或显示简要内容（如果内容较短，直接显示）
7. **可选部分索引**：链接到可选部分文件或显示简要内容（概述、澄清、假设、依赖项）

**关键优化**：
- **功能摘要保留**：单功能Spec的功能摘要必须完整保留在索引文件中，作为导航入口
- **系统级Spec**：通用部分不包含详细内容，只包含链接到common/目录的文件
- **系统级Spec**：功能规格说明只包含模块索引，详细内容在modules/目录的文件中
- **单功能Spec**：用户故事详细内容在user-stories/目录的文件中
- **文件大小控制**：系统级Spec索引文件300-350行，单功能Spec索引文件200-300行

### 阶段7：验证和报告

#### 7.1 验证拆分结果

**通用验证项**：
- ✅ 所有拆分单元文件都已生成
- ✅ 索引文件（spec-index.md）包含完整的索引和导航
- ✅ 索引文件包含"Spec文件清单"部分
- ✅ 所有链接都有效
- ✅ 原始文件（spec.md）保持不变
- ✅ 文件大小合理
- ✅ 文件清单中的文件路径和行数信息准确

**Spec类型特定验证项**：

| 验证项 | 系统级Spec | 单功能Spec |
|--------|-----------|-----------|
| 拆分单元文件 | modules/*.md | user-stories/*.md |
| 通用文件 | common/*.md | requirements.md, success-criteria.md（如果拆分） |
| 索引文件内容 | 索引和导航（不包含通用部分详细内容） | 功能摘要（完整保留）+ 索引和导航 |
| 引用链接类型 | 模块内锚点链接，通用部分文件链接 | 用户故事文件链接 |
| 文件大小限制 | 索引<400行，模块<400行，通用<150行 | 索引<350行，用户故事<200行 |
| 特殊验证 | 引用关系验证 | 功能摘要完整性验证（业务语言，无技术细节） |

#### 7.2 生成拆分报告

**通用报告内容**：
1. **拆分统计**：原始文件行数、拆分后文件数量、各文件行数
2. **文件结构**：生成的目录结构、文件路径列表
3. **文件清单信息**：所有Spec文件列表、映射表、快速导航指引
4. **优化效果**：索引文件精简效果、拆分效果、单次上下文长度优化
5. **后续操作建议**：如何更新Spec、如何使用拆分后的文件、如何识别所有Spec文件

**Spec类型特定报告内容**：

| 报告项 | 系统级Spec | 单功能Spec |
|--------|-----------|-----------|
| **拆分统计** | 索引文件 + 模块文件数 + 通用文件数 | 索引文件 + 用户故事文件数 + 其他拆分文件数 |
| | 索引文件行数（精简版，约300-350行） | 索引文件行数（包含功能摘要，约200-300行） |
| | 各模块文件行数、各通用文件行数 | 各用户故事文件行数、需求/成功标准文件行数 |
| **特殊统计** | 引用关系统计（引用关系数量、列表、验证结果） | 功能摘要验证（完整性、业务语言、包含项） |
| **映射表** | 功能模块映射表（模块名称、文件路径、功能数量、功能ID列表） | 用户故事映射表（用户故事标题、优先级、文件路径） |
| **导航指引** | 按开发任务、按功能ID、按通用规范 | 按优先级、按用户故事 |
| **优化效果** | 从约664行减少到约300-350行；3个独立通用文件；单次加载约400-600行 | 包含功能摘要约200-300行；每个用户故事独立文件；单次加载约300-500行 |
| **特别说明** | - | 功能摘要保留在索引文件中，作为快速理解功能全貌的入口（5分钟内） |

## 引用关系识别规则

**字段类型识别**：根据字段关键词识别引用关系（如"归属部门"→部门管理，"岗位"→岗位管理，"角色"→角色管理，"权限"→菜单管理）

**数据模型识别**：根据外键字段识别引用关系（如"部门ID"→部门表，"角色ID"→角色表，"菜单ID"→菜单表，"岗位ID"→岗位表）

**业务规则识别**：根据业务规则中的描述识别依赖关系（如"必须属于某个部门"→部门管理，"必须分配角色"→角色管理，"权限配置"→菜单管理）

## 使用示例

**示例1**：`/拆分Spec` - 拆分当前目录或docs/specs/下的spec.md文件

**示例2**：`/拆分Spec docs/specs/competitor/spec.md` - 拆分指定路径的Spec文件

**处理流程**：
- **系统级Spec**：验证输入 → 判断Spec类型 → 解析结构 → 识别模块边界 → 识别引用关系 → 生成模块文件 → 生成通用文件 → 生成索引文件 → 验证报告
- **单功能Spec**：验证输入 → 判断Spec类型 → 解析结构 → 识别功能摘要 → 识别用户故事边界 → 生成用户故事文件 → 生成索引文件（包含功能摘要） → 验证报告

## 注意事项

**原始文件保留**：原始spec.md文件保持不变，用于后续更新。更新时直接修改spec.md后重新运行拆分命令。

**引用关系维护**：引用关系由命令自动识别和维护，使用锚点链接。如果识别不准确，可在模块文件中手动调整，重新拆分时会重新识别。

**文件更新流程**：修改spec.md → 重新运行`/拆分Spec` → 自动更新所有拆分文件（modules/*.md、common/*.md、spec-index.md）

**文件大小控制**：
- **系统级Spec**：索引文件300-350行，模块文件100-400行，通用文件50-150行（交互流程、业务规则、UI/UX规范）。如果单个模块过大（>500行），可考虑进一步拆分。
- **单功能Spec**：索引文件200-300行（包含功能摘要），用户故事文件50-200行。如果单个用户故事过大（>300行），可考虑进一步拆分。

**文件清单使用**：开发和测试人员通过spec-index.md的"Spec文件清单"部分快速识别所有Spec文件，包括文件结构、文件列表、模块映射表和快速导航。

## 输出文件结构

**通用结构**：
```
docs/specs/{编号}-{名称}/
├── spec.md                   # 原始Spec文件（保持不变）
├── spec-index.md             # 索引文件
└── [拆分单元目录]/            # modules/（系统级）或 user-stories/（单功能）
```

**系统级Spec完整结构**：
```
docs/specs/{编号}-{模块名}/
├── spec.md                   # 原始Spec文件
├── spec-index.md             # 索引文件（精简版，约300-350行）
├── modules/                  # 功能模块目录
│   └── [模块名].md           # 各模块文件（包含引用关系）
└── common/                   # 通用部分目录
    ├── interaction-flows.md  # 交互流程
    ├── business-rules.md     # 业务规则
    └── ui-ux-specs.md        # UI/UX规范
```

**单功能Spec完整结构**：
```
docs/specs/{编号}-{功能名}/
├── spec.md                   # 原始Spec文件
├── spec-index.md             # 索引文件（包含功能摘要，约200-300行）
├── user-stories/             # 用户故事目录
│   ├── user-story-{N}.md    # 各用户故事文件（按优先级）
│   └── edge-cases.md         # 边缘情况
├── requirements.md           # 需求部分（仅当内容>200行时）
└── success-criteria.md       # 成功标准（仅当内容>100行时）
```

### 工作流程

```
/竞品复刻 → /拆分Spec → [修改spec.md] → /拆分Spec → /原型设计
```
