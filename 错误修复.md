## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于检测并修复当前启动工程的所有页面报错。命令会自动识别当前运行的服务器，使用浏览器（优先Chrome）打开页面进行验证，检测浏览器控制台错误、网络请求失败、页面渲染问题等，并持续修复错误直到所有错误彻底解决。

**核心特性**：
- ✅ **智能工程定位**：自动从对话历史、当前打开文件、工作区上下文等多维度识别当前工程
- ✅ **服务器智能匹配**：自动检测运行的服务器，并匹配到对应的工程路径
- ✅ **多服务器处理**：如果检测到多个运行的服务器，智能匹配或让用户选择
- ✅ **浏览器验证**：使用 `@Browser` 工具（优先Chrome）打开页面进行实际验证
- ✅ **全面错误检测**：检测控制台错误、网络请求失败、页面渲染问题、编译错误等
- ✅ **持续修复循环**：发现问题后自动修复，重新验证，直到所有错误解决
- ✅ **智能修复策略**：根据错误类型采用不同的修复策略（参考错误检测与监控模块）
- ✅ **详细错误报告**：生成详细的错误报告和修复记录

**Browser工具使用**：
- 使用 `mcp_cursor-browser-extension_browser_navigate` 导航到页面
- 使用 `mcp_cursor-browser-extension_browser_console_messages` 获取控制台错误
- 使用 `mcp_cursor-browser-extension_browser_network_requests` 获取网络请求
- 使用 `mcp_cursor-browser-extension_browser_snapshot` 获取页面快照
- 使用 `mcp_cursor-browser-extension_browser_wait_for` 等待页面加载

## 执行流程

### 阶段0：识别当前工程和服务器状态

#### 0.1 识别当前工程路径

**路径识别规则**（按优先级，从高到低）：

1. **用户指定路径**：如果用户提供了路径参数
   - 使用用户指定的路径
   - 验证路径是否存在且为有效工程

2. **从对话历史提取工程路径**：
   - 检查当前对话中是否提到过工程路径
   - 检查是否提到过项目名称（从项目名称推断路径：`docs/frontend-projects/{项目名}-initial` 或 `docs/frontend-projects/{项目名}`）
   - 检查是否提到过Spec目录（从Spec目录推断工程路径）
   - 如果找到，验证路径是否存在

3. **从当前打开的文件推断**：
   - 检查当前打开的文件路径
   - 如果文件在某个工程目录下（包含`package.json`或`.html`），使用该工程目录
   - 如果文件在`docs/specs/{编号}-{名称}/`目录下，推断对应的工程路径：
     - 优先：`docs/frontend-projects/{名称}-initial`
     - 备选：`docs/frontend-projects/{名称}`
   - 验证推断的路径是否存在

4. **从工作区上下文推断**：
   - **如果当前工作目录在某个Spec目录下**：
     - 自动识别Spec目录：`docs/specs/{编号}-{名称}/`
     - 从目录名称提取项目名称（去掉编号前缀）
     - 推断工程路径：`docs/frontend-projects/{名称}-initial` 或 `docs/frontend-projects/{名称}`
     - 示例：`docs/specs/018-message-subscription-config/` → `docs/frontend-projects/message-subscription-config-initial`
   - **如果当前工作目录在某个工程目录下**：
     - 检查是否包含`package.json`（前端项目）
     - 检查是否包含`.html`文件（HTML原型）
     - 如果符合，使用当前工作目录

5. **从最近访问的文件推断**：
   - 检查最近访问的文件（最近5-10个文件）
   - 如果文件在某个工程目录下，使用该工程目录
   - 如果找到多个候选工程，进入步骤6

6. **搜索项目目录**：
   - 搜索`docs/frontend-projects/`目录下的项目
   - 搜索`docs/prototypes/`目录下的原型
   - 如果找到多个候选工程，列出让用户选择

**工程验证**：
- 验证路径是否存在
- 验证是否为有效工程（包含`package.json`或`.html`文件）
- 如果验证失败，尝试下一个优先级规则

#### 0.2 检测服务器运行状态并匹配工程

**检测步骤**：

1. **检测所有运行的服务器**：
   ```bash
   # 检测常见开发服务器端口
   # Vite: 5173, Vue CLI: 8080, React: 3000, 静态服务器: 8000, 8090等
   lsof -i :5173 -i :8080 -i :3000 -i :8000 -i :8090 -i :5174 -i :8081
   ```

2. **从进程检测**：
   ```bash
   # 检测Node.js进程，提取工作目录
   ps aux | grep -E "node|npm|yarn|pnpm|vite|webpack" | grep -v grep
   ```

3. **匹配服务器与工程路径**：
   - **如果已确定工程路径**：
     - 检查是否有服务器运行在该工程目录下
     - 从进程的工作目录（cwd）匹配工程路径
     - 如果匹配成功，使用该服务器
     - 如果未匹配到，检查是否有服务器运行在工程路径的父目录或子目录
   
   - **如果检测到多个运行的服务器**：
     - 列出所有运行的服务器（端口、进程ID、工作目录）
     - 尝试匹配每个服务器的工作目录与候选工程路径
     - 如果只有一个匹配，使用该服务器
     - 如果有多个匹配或无法确定，让用户选择：
       ```
       ⚠️ 检测到多个运行的服务器：
       
       1. 端口 5173 - Vite - 工作目录: /path/to/project1
       2. 端口 8080 - Vue CLI - 工作目录: /path/to/project2
       3. 端口 3000 - React - 工作目录: /path/to/project3
       
       请选择要修复的服务器（输入序号）：
       ```
   
   - **如果只检测到一个服务器**：
     - 验证服务器工作目录是否与工程路径匹配
     - 如果匹配，使用该服务器
     - 如果不匹配，提示用户：
       ```
       ⚠️ 检测到服务器运行在：{服务器工作目录}
       但当前工程路径为：{工程路径}
       
       请选择：
       1. 使用检测到的服务器（{服务器工作目录}）
       2. 启动新服务器（{工程路径}）
       3. 取消
       ```

4. **识别服务器类型和端口**：
   - 从匹配的服务器中提取端口号
   - 识别服务器类型（Vite/Vue CLI/React/静态服务器）
   - 确定访问URL

5. **如果服务器未运行或未匹配到**：
   - 如果已确定工程路径，进入阶段1：启动服务器
   - 如果未确定工程路径，先完成工程路径识别

#### 0.3 确定访问URL

**URL确定规则**：

- **如果已匹配到运行中的服务器**：
  - 使用匹配的服务器端口号
  - URL格式：`http://localhost:{端口号}`
  - 记录服务器信息（端口、类型、工作目录）用于后续验证

- **如果服务器未运行，需要启动新服务器**：
  - **如果是前端项目**：
    - 检查配置文件中的端口（`vite.config.ts`、`vue.config.js`等）
    - 如果未配置，使用默认端口（Vite: 5173, Vue CLI: 8080, React: 3000）
    - URL格式：`http://localhost:{端口号}`
  
  - **如果是HTML原型**：
    - 默认端口：8000
    - URL格式：`http://localhost:8000/index.html` 或 `http://localhost:8000/`

**工程路径确认**：
- 在确定URL之前，确保工程路径已正确识别
- 如果工程路径不确定，提示用户确认：
  ```
  📋 已识别工程路径：{工程路径}
  
  请确认是否正确：
  1. 是，继续修复
  2. 否，重新指定路径
  ```

**执行方式**：
- 使用 `read_file` 工具读取 `.cursor/commands/company/设计预览.md` 文件

**执行内容**：

**完成后**：
- 记录服务器URL和端口号
- 进入阶段2：浏览器验证

### 阶段2：浏览器验证与错误检测

#### 2.1 打开浏览器并加载页面

**使用Browser工具打开页面**：

1. **优先使用Chrome浏览器**：
   - 使用 `@Browser` 工具（`mcp_cursor-browser-extension_browser_navigate`）导航到页面URL
   - 如果Chrome不可用，使用系统默认浏览器
   - URL格式：`http://localhost:{端口号}` 或用户指定的URL

2. **等待页面加载完成**：
   - 使用 `browser_wait_for` 等待页面关键元素加载
   - 等待时间：至少3-5秒，确保页面完全加载
   - 检测页面标题或关键DOM元素出现
   - 如果页面加载超时，记录为错误

3. **捕获页面快照**：
   - 使用 `browser_snapshot` 获取页面可访问性快照
   - 用于后续错误检测和DOM元素检查
   - 保存快照用于对比修复前后的状态

#### 2.2 检测浏览器控制台错误

**检测步骤**：

1. **获取控制台消息**：
   - 使用 `browser_console_messages` 获取所有控制台消息
   - 过滤错误和警告消息

2. **错误类型识别**：
   - **JavaScript运行时错误**：
     - 检测：`Error`、`TypeError`、`ReferenceError`、`SyntaxError`
     - 提取：错误消息、文件名、行号、堆栈信息
   
   - **资源加载错误**：
     - 检测：`Failed to load resource`、`404`、`500`、`CORS`
     - 提取：资源URL、错误类型、状态码
   
   - **警告信息**：
     - 检测：`Warning`、`DeprecationWarning`
     - 记录警告但不作为阻塞错误

3. **错误分类和记录**：
   - 将错误分类为：**阻塞错误**（必须修复）和**警告**（可选修复）
   - 记录每个错误的详细信息

#### 2.3 检测网络请求失败

**检测步骤**：

1. **获取网络请求**：
   - 使用 `browser_network_requests` 获取所有网络请求
   - 过滤失败的请求（状态码 >= 400）

2. **错误类型识别**：
   - **404错误**：资源文件不存在
   - **500错误**：服务器内部错误
   - **CORS错误**：跨域请求被阻止
   - **超时错误**：请求超时
   - **其他HTTP错误**：4xx、5xx状态码

3. **提取错误详情**：
   - 请求URL
   - 请求方法（GET/POST等）
   - 响应状态码
   - 错误消息

#### 2.4 检测页面渲染问题

**检测步骤**：

1. **检查关键元素**：
   - 使用 `browser_snapshot` 检查关键DOM元素是否存在
   - 检查页面标题、主要内容区域、导航栏等

2. **检查样式问题**：
   - 检测样式文件是否加载成功
   - 检查是否有明显的样式错误（布局错乱、颜色异常等）

3. **检查功能问题**：
   - 检测交互元素是否可点击
   - 检查表单元素是否存在
   - 检查关键功能是否可用

#### 2.5 检测服务器端错误

**执行模块**：`.cursor/modules/错误检测与监控.md` 的4.1节

请读取并执行该模块中的服务器端错误检测步骤，包括：
- 端口冲突检测
- 编译错误检测
- 依赖缺失检测
- 配置错误检测
- 环境变量缺失检测

**检测方式**：
- 如果服务器在后台运行，持续监控服务器输出日志
- 检测服务器启动和运行过程中的各种错误

### 阶段3：错误修复循环

#### 3.1 错误汇总与优先级排序

**错误汇总**：

收集所有检测到的错误：
- 浏览器控制台错误
- 网络请求失败
- 页面渲染问题
- 服务器端错误（编译错误、依赖缺失等）

**优先级排序**（按修复顺序）：

1. **服务器端阻塞错误**（最高优先级）：
   - 编译错误（代码无法编译）
   - 依赖缺失（模块无法加载）
   - 端口冲突（服务器无法启动）
   - 配置错误（配置文件问题）

2. **资源加载错误**：
   - 404错误（文件不存在）
   - 500错误（服务器错误）
   - CORS错误（跨域问题）

3. **JavaScript运行时错误**：
   - 未定义变量/函数
   - 类型错误
   - 语法错误

4. **页面渲染问题**：
   - 关键元素缺失
   - 样式加载失败
   - 布局问题

5. **警告信息**（最低优先级）：
   - 控制台警告
   - 废弃API警告

#### 3.2 执行错误修复

**⚠️ 重要**：**必须先加载并完全遵循**模块文件：`.cursor/modules/错误检测与监控.md`

**执行模块**：`.cursor/modules/错误检测与监控.md` 的4.2节

请读取并执行该模块中的错误修复策略，包括：
- 按优先级自动修复可修复的错误
- 端口冲突、依赖缺失、环境变量缺失、配置文件缺失等服务器端错误的修复
- 常见语法错误、导入路径错误、类型错误的修复
- 浏览器端错误的修复（JavaScript运行时错误、网络请求失败、页面渲染问题）

**修复执行**：
- 根据错误优先级，依次执行修复策略
- 对于服务器端错误，严格按照模块4.2节的修复策略执行
- 对于浏览器端错误，参考模块4.2节第8条的修复策略执行
- 记录每次修复的操作和结果

#### 3.3 重新验证修复结果

**验证步骤**：

1. **等待修复完成**：
   - 如果是服务器端修复，等待服务器重启（检测到"Local:"、"http://"或"ready"输出）
   - 如果是代码修复，等待编译完成（检测到编译成功输出）
   - 等待时间：至少3-5秒，确保修复生效

2. **刷新浏览器页面**：
   - 使用 `browser_navigate` 重新导航到页面URL（完全重新加载）
   - 或使用 `browser_press_key` 按F5刷新页面
   - 等待页面重新加载完成（使用 `browser_wait_for`）

3. **重新检测错误**：
   - 重复阶段2的所有检测步骤：
     - 获取控制台消息（`browser_console_messages`）
     - 获取网络请求（`browser_network_requests`）
     - 获取页面快照（`browser_snapshot`）
   - 检查之前发现的错误是否已修复
   - 检查是否有新错误出现
   - 对比修复前后的错误列表

4. **记录修复结果**：
   - 记录已修复的错误（从错误列表中移除）
   - 记录仍存在的错误（标记为需要继续修复）
   - 记录新出现的错误（添加到错误列表）
   - 更新错误统计信息

#### 3.4 修复循环控制

**循环条件**：

- **继续修复**：如果仍有阻塞错误或可自动修复的错误
- **停止修复**：如果所有错误已修复，或只剩下需要用户手动修复的错误

**循环流程**：
```
检测错误 → 汇总错误 → 按优先级修复 → 重新验证 → 检查是否还有错误
    ↑                                                      ↓
    └─────────────────── 如果还有错误，继续循环 ───────────┘
```

**终止条件**：

1. ✅ **所有错误已修复** → **成功，告知用户**
2. ❌ **无法自动修复的错误** → **暂停，显示错误详情，询问用户是否继续**
3. ⏹️ **用户手动终止** → **停止修复**

**最大修复次数**：
- 设置最大修复循环次数：**20次**
- 如果达到最大次数仍有错误，停止并报告

## 命令参数说明

```
/错误修复 [选项] [路径]

选项:
  --url=<url>             指定页面URL（如果服务器已运行）
  --port=<port>           指定服务器端口（如果服务器未运行）
  --project=<path>         指定工程路径（覆盖自动检测）
  --list-servers           列出所有运行的服务器，让用户选择
  --max-iterations=<n>    最大修复循环次数（默认：20）
  --no-browser            不使用浏览器验证（仅检测服务器端错误）
  --fix-only              仅修复，不生成详细报告

路径:
  如果未指定路径，按以下优先级自动检测：
  1. 对话历史中提到的工程路径
  2. 当前打开文件所在的工程目录
  3. 当前工作目录（如果在工程目录或Spec目录下）
  4. 最近访问的文件所在的工程目录
  5. 搜索 docs/frontend-projects/ 和 docs/prototypes/ 目录

示例:
  /错误修复                                    # 自动检测并修复当前工程
  /错误修复 ./my-project                      # 修复指定项目
  /错误修复 --project=./my-project            # 明确指定工程路径
  /错误修复 --url=http://localhost:5173      # 修复指定URL的页面
  /错误修复 --port=8080                      # 在指定端口启动并修复
  /错误修复 --list-servers                   # 列出所有服务器，让用户选择
  /错误修复 --max-iterations=10              # 最多修复10次循环
```

## 注意事项

1. **工程定位**：
   - 命令会智能识别当前工程，优先级：用户指定 > 对话历史 > 当前打开文件 > 工作区上下文 > 最近访问文件
   - 如果当前在Spec目录下，会自动推断对应的工程路径（`docs/frontend-projects/{名称}-initial`）
   - 如果无法确定工程，会列出候选工程让用户选择
   - 可以通过`--project`参数明确指定工程路径

2. **服务器匹配**：
   - 如果检测到多个运行的服务器，会尝试匹配服务器工作目录与工程路径
   - 如果只有一个匹配，自动使用该服务器
   - 如果有多个匹配或无法确定，会列出让用户选择
   - 如果服务器未运行，会自动启动服务器
   - 修复过程中可能会重启服务器

3. **浏览器要求**：
   - 优先使用Chrome浏览器进行验证
   - 如果Chrome不可用，使用系统默认浏览器
   - 浏览器需要支持开发者工具和控制台

4. **修复限制**：
   - 只能自动修复常见错误
   - 复杂错误需要用户手动修复
   - 某些错误可能需要用户提供额外信息

5. **修复循环**：
   - 默认最多修复20次循环
   - 如果达到最大次数仍有错误，会停止并报告
   - 可以通过`--max-iterations`参数调整

6. **性能影响**：
   - 修复过程可能会重启服务器
   - 浏览器验证会打开新标签页
   - 大量错误可能需要较长时间修复
