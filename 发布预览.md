## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于在ACW上创建版本并发布预览。命令会自动检测Git仓库信息，执行git提交和推送操作，根据Git地址生成预览URL，并询问是否在Chrome浏览器中打开预览地址。

**核心特性**：
- ✅ 支持前端工程和原型目录两种项目类型
- ✅ 自动检测Git仓库地址
- ✅ 自动执行git commit和git push（自动生成提交信息）
- ✅ 根据Git地址生成ACW预览URL
- ✅ 指导用户在ACW上创建版本
- ✅ 询问是否在Chrome浏览器中打开预览地址

**Git地址与预览URL**：
- Git地址格式：`http://gitlab.bangdao-tech.com/{git用户名}/{ACW平台用户名}-{应用名}.git`
- 预览URL格式：`https://acw.bangdao-tech.com/rp/{应用名}`
- 示例：`http://gitlab.bangdao-tech.com/68597/liuhuanwei-COMMON_ADMIN.git` → `https://acw.bangdao-tech.com/rp/COMMON_ADMIN`

## 执行流程

### 阶段0：项目识别与定位

1. **识别项目路径**：
   - 如果用户指定了 `--project-path=<path>` 参数，使用指定路径
   - 否则按优先级自动识别：
     1. 最近对话中涉及的工程或原型路径
     2. `docs/frontend-projects/` 目录下最新的工程（包括 `-initial` 工程）
     3. `docs/prototypes/` 目录下最新的原型（自动查找最新版本）
   - 如果都未找到，提示用户指定路径

2. **确定Git仓库根目录**：
   - 从项目路径向上查找`.git`目录
   - 记录：`PROJECT_PATH = "{项目路径}"`，`GIT_ROOT = "{Git仓库根目录}"`
   - 如果找不到`.git`目录，后续需要用户提供Git信息

3. **识别项目类型**：
   - **原型目录**：路径匹配 `docs/prototypes/{项目名}/{模块名}`，存在版本目录（`v1/`、`v2/`等）
   - **前端工程**：路径匹配 `docs/frontend-projects/{项目名}`，存在 `package.json`、`vite.config.ts` 等
   - 记录：`PROJECT_TYPE = "prototype"` 或 `"frontend"`

---

### 阶段1：检测Git仓库信息

1. **检查Git仓库**：
   ```bash
   cd {GIT_ROOT}
   test -d .git
   git remote get-url origin
   git branch --show-current
   ```

2. **如果不是Git仓库或无法获取**：
   - ⚠️ **必须让用户填写项目在ACW上的Git地址**
   - 提示用户提供：Git地址、用户名、密码
   - 验证Git地址格式（需包含`gitlab.bangdao-tech.com`）
   
   **提示用户格式**：
   ```
   ⚠️ 无法自动获取Git仓库信息
   
   请提供项目在ACW上的Git地址：
   1. 项目Git地址：________________
   2. Git用户名：________________
   3. Git密码：________________
   
   示例：
   - Git地址：http://gitlab.bangdao-tech.com/68597/liuhuanwei-COMMON_ADMIN.git
   - 用户名：liuhuanwei
   ```

---

### 阶段2：提取项目名称并生成预览URL

从Git地址提取应用名：
- 提取仓库名称（去掉`.git`）：`liuhuanwei-COMMON_ADMIN`
- 去掉ACW平台用户名前缀：`COMMON_ADMIN`
- 生成预览URL：`https://acw.bangdao-tech.com/rp/COMMON_ADMIN`

---

### 阶段3：构建与提交

**⚠️ 重要**：根据项目类型执行不同的处理流程。

#### 3.1 前端工程：Vite构建

1. **验证Vite配置**：
   - 检查 `vite.config.ts` 的 `base` 配置
   - ✅ 正确：`const base = mode === 'production' ? './' : '/'`（动态配置）
   - ❌ 错误：`base: '/'` 或 `base: './'`（固定值）
   - 如果配置错误，自动修复为动态配置
   - 检查 `index.html` 中的 favicon 路径是否为相对路径

2. **执行生产构建**：
   ```bash
   NODE_ENV=production npm run build
   ```
   - 必须设置 `NODE_ENV=production`，确保Vite使用production模式
   - 如果构建失败，显示错误信息并停止执行

3. **验证构建产物**：
   - 检查 `dist/index.html` 中的资源路径
   - ✅ 正确：`src="./assets/xxx.js"`、`href="./assets/xxx.css"`
   - ❌ 错误：`src="/assets/xxx.js"`、`href="/assets/xxx.css"`

#### 3.2 原型目录：查找dist目录

1. **查找最新版本**：
   - 扫描版本目录（`v1/`、`v2/`、`v3` 等），按版本号排序
   - 选择版本号最大的版本目录
   - 验证该版本目录下是否存在 `dist/` 目录
   - 记录：`LATEST_VERSION = "v{版本号}"`，`DIST_PATH = "{原型目录}/{最新版本}/dist"`

2. **验证dist目录内容**：
   - 检查 `dist/index.html` 是否存在
   - 检查资源文件（CSS、JS、图片等）是否存在

#### 3.3 Git提交和推送

1. **确定目标分支**：
   - **⚠️ 强制使用 `master` 分支**（忽略 `--branch` 参数）
   - 如果当前分支不是 `master`，切换到 `master`：`git checkout master`
   - 如果 `master` 不存在，创建并切换：`git checkout -b master`

2. **暂存更改**（在Git仓库根目录执行）：
   
   **前端工程**：
   - 如果前端工程路径 = Git仓库根目录：`git add .`
   - 如果前端工程路径 ≠ Git仓库根目录：`git add {前端工程相对路径}/`
   - 如果 `dist` 在 `.gitignore` 中：`git add -f {前端工程路径}/dist/`
   
   **原型目录**：
   - **⚠️ 重要**：将dist目录和README.md复制到Git仓库根目录，然后提交根目录下的文件
   - 执行步骤：
     ```bash
     # 复制文件到根目录
     cp -r {原型目录相对路径}/{最新版本}/dist .
     cp {原型目录相对路径}/{最新版本}/README.md .
     # 只提交根目录下的dist和README.md
     git add dist/
     git add README.md
     ```
   - 如果 `dist` 在 `.gitignore` 中：`git add -f dist/`
   - **禁止使用 `git add .`**，禁止提交原型目录路径（如 `docs/prototypes/`）

3. **自动生成提交信息并提交**：
   - **分析更改内容**（在Git仓库根目录执行）：
     ```bash
     git diff --cached  # 已暂存的更改
     git diff           # 未暂存的更改
     ```
   - **生成提交信息**：
     - 识别文件类型和修改类型（`.vue`、`.ts`、`.js`、`.html`、`.md`等）
     - 格式：`feat: {简要描述}` 或 `docs: {描述}` 或 `fix: {描述}`
     - 示例：`feat: 更新用户管理页面`、`docs: 更新需求文档`、`fix: 修复表单验证问题`
     - 如果无法生成，使用：`chore: 更新项目内容 - {当前日期时间}`
   - **提交更改**：
     ```bash
     git commit -m "{自动生成的提交信息}"
     ```
   - **提交信息格式**：
     - **前端工程**：`feat: 更新前端页面` 或 `chore: 更新项目内容`
     - **原型目录**：`feat: 发布原型 {版本号}` 或 `chore: 更新原型 {版本号}`

4. **推送到远程**：
   ```bash
   git push origin master
   ```
   - 如果推送失败（远程有更新），先执行：`git pull --rebase origin master`，然后再次推送

#### 3.4 验证提交范围

**⚠️ 重要**：确保提交的文件范围正确。

1. **验证暂存的文件列表**（提交前）：
   ```bash
   git diff --cached --name-only
   ```
   - **原型目录**：应该只包含 `dist/` 和 `README.md`（在仓库根目录）

2. **验证提交的文件列表**（提交后）：
   ```bash
   git show --name-only HEAD
   ```
   - 检查是否只包含应该提交的文件

3. **验证远程仓库的文件列表**（推送后）：
   ```bash
   git fetch origin
   git ls-tree -r --name-only origin/master
   ```
   - **原型目录**：应该只包含 `.gitignore`（如果存在）、`dist/`、`README.md`（都在仓库根目录）
   - 如果发现其他文件（如 `docs/prototypes/`、`.cursor/`、`docs/specs/` 等），需要清理：
     ```bash
     git rm -r --cached docs/ .cursor/ .specify/ .vscode/ 2>/dev/null || true
     git add dist/ README.md
     git commit -m "chore: 清理仓库，只保留根目录下的dist和README.md"
     git push origin master
     ```

---

### 阶段4：构建产物验证

1. **检查资源路径**：
   ```bash
   grep -E '(src|href)=' {DIST_PATH}/index.html
   ```
   - ✅ 正确：`src="./assets/xxx.js"`、`href="./assets/xxx.css"`
   - ❌ 错误：`src="/assets/xxx.js"`、`href="/assets/xxx.css"`

2. **如果发现路径错误**（仅前端工程）：
   - 重新检查并修复 `vite.config.ts` 配置
   - 确保构建时设置了 `NODE_ENV=production`
   - 重新执行构建

---

### 阶段5：ACW版本创建与预览

1. **提示用户创建ACW版本**：
   ```
   📋 请在ACW上创建版本
   
   请参考语雀文档《ACW原型应用发布指南》：
   https://bd-tech.yuque.com/vy9as8/ceft34/dexkxx76fmhbowiy
   
   操作步骤：
   1. 登录ACW平台
   2. 找到项目：{PROJECT_NAME}
   3. 创建新版本
   4. 填写版本信息（版本号、描述等）
   5. 确认创建
   
   创建完成后，请告诉我是否已成功创建版本。
   ```

   **等待用户确认**：
   - 如果用户回复"已创建"、"完成"、"ok"等 → 继续执行后续步骤
   - 如果用户回复"未创建"、"稍后"等 → 提示用户创建后再继续
   - 如果用户未回复 → 等待3-5秒后继续（假设用户已创建）

2. **显示预览信息**：
   ```
   🎉 发布预览准备完成！
   
   📦 项目信息：
   - 项目类型：{PROJECT_TYPE}
   - 项目名称：{PROJECT_NAME}
   - Git地址：{Git地址}
   - 预览URL：{预览URL}
   {如果是原型目录，显示：- 原型版本：{LATEST_VERSION}}
   
   ✅ Git操作：
   - 提交：{提交信息}
   - 推送：{推送状态}
   ```

3. **询问是否打开预览地址**：
   ```
   🌐 是否在浏览器中打开预览地址？
   
   预览地址：{预览URL}
   
   选项：是/否 或 y/n
   ```
   - 用户选择"是"或"y" → 使用Chrome浏览器打开：
     - **Mac**：`open -a "Google Chrome" "{预览URL}"`
     - **Windows**：`start chrome "{预览URL}"`
     - **Linux**：`google-chrome "{预览URL}"` 或 `xdg-open "{预览URL}"`
   - 用户选择"否"或"n" → 跳过打开浏览器步骤
   - 用户未明确选择 → 默认选择"是"
   - 如果Chrome不可用，使用系统默认浏览器（Mac: `open`，Windows: `start`，Linux: `xdg-open`）

## 命令参数说明

```
/发布预览 [选项] [项目路径]

选项:
  --project-path=<path> 指定前端工程或原型目录路径（如果未指定，会自动识别）
  --no-git             跳过Git操作（不执行commit和push）
  --no-open            不询问是否打开浏览器
  --commit-msg=<msg>    指定Git提交信息（覆盖自动生成）
  --branch=<branch>     指定Git分支（⚠️ 已忽略：强制使用master分支）

项目路径:
  如果未指定，会自动识别工程或原型路径（不询问用户）：
  1. 优先：最近对话中涉及的工程或原型路径
  2. 其次：docs/frontend-projects/ 目录下最新的工程（包括 -initial 工程）
  3. 再次：docs/prototypes/ 目录下最新的原型
  4. 如果都识别不到：询问用户指定工程或原型路径

示例:
  /发布预览                                    # 自动识别工程或原型路径
  /发布预览 --project-path=./my-project       # 指定项目路径
  /发布预览 docs/frontend-projects/my-project  # 指定前端工程路径
  /发布预览 docs/prototypes/my-project/my-module  # 指定原型目录路径
  /发布预览 --no-git                          # 跳过Git操作
  /发布预览 --no-open                         # 不打开浏览器
  /发布预览 --commit-msg="feat: 更新功能"     # 指定提交信息
```

## 注意事项

1. **Git提交范围**：
   - **前端工程**：如果前端工程是Git仓库根目录，提交整个工程；如果是子目录，只提交该工程目录
   - **原型目录**：**将最新版本的dist目录和README.md复制到Git仓库根目录，然后只提交根目录下的 `dist/` 和 `README.md`**，**不会提交原型目录路径（如 `docs/prototypes/`）或其他文件**
   - **根目录结构**：提交后，远程仓库根目录应该直接显示 `dist/` 目录和 `README.md` 文件

2. **强制推送到 `master` 分支**：
   - 无论用户是否指定 `--branch` 参数，都强制推送到 `master` 分支
   - 如果当前分支不是 `master`，会自动切换到 `master` 分支

3. **ACW版本创建**：
   - 需要在ACW平台上手动创建版本（参考语雀文档）
   - 创建版本后才能访问预览地址

4. **预览URL生成**：
   - 预览URL根据Git地址自动生成
   - 如果Git地址格式不符合预期，可能无法正确生成预览URL
