## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于在原型或前端页面发生细微调整后，自动识别变化并更新功能规格文档。更新前会生成详细的更新清单供用户确认，确保更新的准确性和可控性。

**适用场景**：
- 原型页面调整后需要同步更新Spec
- 前端页面实现后需要补充或修正Spec
- 基于新的UI截图更新Spec内容
- 根据在线原型链接更新Spec

**核心特性**：
- ✅ **智能差异识别**：自动对比现有Spec与新内容，识别需要更新的部分
- ✅ **更新清单确认**：生成详细的更新清单，用户确认后再执行更新
- ✅ **灵活确认方式**：支持同意、取消、自定义回复
- ✅ **保留历史记录**：在Spec中记录更新历史，便于追溯
- ✅ **精确更新**：只更新变化的部分，保持其他内容不变

## 执行流程

### 阶段0：输入识别与文件定位

#### 0.1 识别输入类型

**检测输入类型**（按优先级）：
1. 以 `http://` 或 `https://` 开头 → **在线原型/页面URL**
2. 路径特征（`/` 或 `./` 或盘符）且指向文件夹且包含.html文件 → **HTML文件夹**
3. 以`.html`结尾 → **单HTML文件**
4. 对话中包含图片 → **图片更新**
5. 否则 → 提示用户提供有效的输入源

#### 0.2 定位Spec文件

**查找策略**：
1. 如果用户指定了Spec文件路径：使用指定路径
2. 如果未指定：在当前目录或`docs/specs/`目录下查找`spec.md`文件
3. 如果找到多个spec.md文件：列出所有文件让用户选择

**验证规则**：
- ✅ Spec文件必须存在
- ✅ Spec文件必须是Markdown格式
- ✅ Spec文件必须包含基本结构（功能摘要、用户场景、需求等）

#### 0.3 确认更新范围

**询问用户**：
- 是否只更新特定部分（如：仅更新表单字段、仅更新用户故事等）
- 如果用户未指定，则进行全量对比更新

### 阶段1：提取新内容

#### 1.1 根据输入类型提取内容

**在线原型/页面URL**：
- **⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/在线网页解析.md`
- **加载方式**：使用 `read_file` 工具读取该模块文件，理解完整的执行流程
- **执行要求**：严格按照模块中定义的7个阶段执行，包括：
  * 阶段2.5：页面滚动检查与完整内容提取（⚠️ 必须执行）
  * 阶段3：UI元素提取
  * 阶段5：推断功能需求
- **输出**：提取的功能描述、UI元素、表单字段、业务规则等

**HTML文件夹/单HTML文件**：
- 读取HTML文件内容
- 提取UI元素（表格、表单、按钮、筛选器等）
- 识别页面类型和功能
- 提取业务规则和字段信息

**图片更新**：
- 识别页面类型（列表页、表单页、详情页等）
- 提取UI元素（表格列、表单字段、按钮等）
- 提取字段属性（字段名称、类型、必填性等）
- 提取交互需求

#### 1.2 结构化提取结果

**提取内容结构**：
- **功能需求**：功能点列表（FR格式）
- **用户故事**：用户故事列表（US格式）
- **表单字段**：字段名称、类型、必填性、验证规则
- **表格列**：列名、数据类型、操作列
- **交互元素**：按钮、筛选器、分页器等
- **业务规则**：验证规则、约束条件、操作流程
- **数据实体**：实体名称、属性、关系

### 阶段2：对比分析与差异识别

#### 2.1 读取现有Spec内容

**提取现有Spec结构**：
- 功能摘要（核心功能概览、关键业务流程、关键数据实体、关键交互点）
- 用户故事（标题、优先级、描述、验收场景）
- 功能需求（FR-XXX格式的需求列表）
- 关键实体（实体名称、属性、关系）
- 成功标准（SC-XXX格式的标准列表）

#### 2.2 智能对比分析

**对比维度**：

1. **功能需求对比**：
   - 新增需求：新内容中有，现有Spec中没有
   - 删除需求：现有Spec中有，新内容中没有（标记为待确认）
   - 修改需求：需求描述发生变化
   - 未变化需求：保持一致的需求

2. **用户故事对比**：
   - 新增用户故事：新内容中有，现有Spec中没有
   - 修改用户故事：用户故事描述、验收场景发生变化
   - 删除用户故事：现有Spec中有，新内容中没有（标记为待确认）

3. **表单字段对比**：
   - 新增字段：新内容中有，现有Spec中没有
   - 删除字段：现有Spec中有，新内容中没有（标记为待确认）
   - 修改字段：字段类型、必填性、验证规则发生变化
   - 字段顺序变化：字段顺序调整

4. **表格列对比**：
   - 新增列：新内容中有，现有Spec中没有
   - 删除列：现有Spec中有，新内容中没有（标记为待确认）
   - 修改列：列名、数据类型、操作列变化

5. **交互元素对比**：
   - 新增按钮/操作：新内容中有，现有Spec中没有
   - 删除按钮/操作：现有Spec中有，新内容中没有（标记为待确认）
   - 修改操作：操作描述、流程变化

6. **业务规则对比**：
   - 新增规则：新内容中有，现有Spec中没有
   - 修改规则：规则描述发生变化
   - 删除规则：现有Spec中有，新内容中没有（标记为待确认）

#### 2.3 生成差异报告

**差异分类**：
- **新增项**：需要添加到Spec的内容
- **修改项**：需要更新Spec中现有内容
- **删除项**：需要从Spec中移除的内容（需用户确认）
- **未变化项**：保持不变的内容（仅用于报告）

**差异标记**：
- 每个差异项包含：类型（新增/修改/删除）、位置（Spec中的章节）、原内容、新内容、影响范围

### 阶段4：用户确认处理

#### 4.1 解析用户回复

2. **取消**：
   - 关键词：`取消`、`不更新`、`no`、`n`、`✗`、`❌`
   - 行为：终止更新，不修改Spec文件

3. **自定义回复**：
   - 包含具体指令（如："只执行1.1"、"跳过删除项"、"修改FR-001为..."）

#### 4.2 处理自定义回复

### 阶段5：版本管理（在执行Spec更新之前）

#### 5.1 版本管理初始化

**⚠️ 必须执行**：在继续之前，**必须先加载并完全遵循**模块文件：`.cursor/modules/spec-version-management.md`

**执行步骤**：
1. 读取当前版本号（从 `FEATURE_DIR/VERSION` 文件，如果不存在则默认为 v1.0.0）
2. 计算新版本号（Minor版本递增，如：v1.2.0 → v1.3.0）
3. 保存旧版本到 `FEATURE_DIR/versions/{旧版本号}-spec.md`
4. 更新 `FEATURE_DIR/VERSION` 文件为新版本号
5. 准备CHANGELOG记录：
   - 变更类型：🔄 更新Spec
   - 变更原因：基于原型/前端页面调整更新Spec
   - 变更内容：从更新清单中提取变更项
   - 影响范围：从更新清单中提取影响范围
   - 验证状态：✅ 已通过更新Spec验证

### 阶段6：执行Spec更新

#### 6.1 应用更新项

**更新策略**：

1. **新增项**：
   - 在Spec的相应位置插入新内容
   - 保持编号连续性（FR-XXX、US-X等）
   - 保持格式一致性

2. **修改项**：
   - 定位到Spec中的具体位置
   - 替换原内容为新内容
   - 保留格式和结构

3. **删除项**（用户确认后）：
   - 从Spec中移除指定内容
   - 调整后续编号（如删除FR-002后，原FR-003变为FR-002）
   - 更新相关引用

#### 6.2 更新Spec文件

**更新操作**：
1. 读取现有Spec文件
2. 按照更新清单逐项应用更新
3. **更新文档头部信息**：
   - 更新 `**当前版本**` 字段为新版本号
   - 更新 `**最后更新**` 字段为当前日期
   - 在版本信息章节添加新版本记录
4. 添加更新历史记录（在Spec末尾或单独章节，可选，因为已有CHANGELOG.md）

**更新历史记录格式**：

```markdown
## 更新历史

### [更新日期] - 更新来源：[输入类型]
- **更新项数**：新增X项，修改Y项，删除Z项
- **更新来源**：[URL/路径]
- **更新摘要**：[简要说明]
- **详细变更**：
  - 新增：[列出新增项]
  - 修改：[列出修改项]
  - 删除：[列出删除项]
```

#### 6.3 更新CHANGELOG.md

**更新操作**：
1. 在 `FEATURE_DIR/CHANGELOG.md` 文件开头（`---` 之后）添加新版本记录
2. 包含完整的变更信息（变更类型、原因、内容、影响范围、验证状态）

#### 6.4 验证更新结果

**验证项**：
- ✅ 所有更新项都已应用
- ✅ Spec文件格式正确
- ✅ 编号连续性正确
- ✅ 引用关系正确
- ✅ 文件可读性良好

#### 6.1 生成更新报告

**报告内容**：

## 使用示例

### 示例1：基于在线原型更新Spec

**输入**：`/更新Spec https://axure.cloud/xxx/prototype`

**处理流程**：
1. 识别为在线原型URL
2. 使用在线网页解析模块提取新内容

4. 生成更新清单
5. 用户确认后执行更新

**输入**：`/更新Spec ./prototypes/user-list.html`

**处理流程**：

3. 对比现有Spec，识别差异
4. 生成更新清单
5. 用户确认后执行更新

### 示例3：基于图片更新Spec

**输入**：`/更新Spec [上传UI截图]`

**处理流程**：
1. 识别为图片输入
2. 提取图片中的UI元素
3. 对比现有Spec，识别差异
4. 生成更新清单
5. 用户确认后执行更新

### 示例4：自定义更新范围

**输入**：`/更新Spec https://example.com/prototype --only=form-fields`

2. 提取新内容

4. 生成仅包含表单字段更新的清单
5. 用户确认后执行更新

- 所有删除项都需要用户明确确认
- 删除项可能影响其他功能，需要谨慎处理

### 3. 编号连续性
- 更新后需要确保编号连续性（FR-XXX、US-X等）
- 删除项后需要重新编号

### 4. 引用关系
- 更新后需要检查引用关系是否正确
- 如果使用了`/拆分Spec`命令，可能需要重新拆分

### 5. 更新历史
- 每次更新都会在Spec中记录更新历史
- 更新历史有助于追溯变更原因

### 6. 性能考虑
- 大型Spec文件更新可能需要较长时间
- 建议分批更新或使用`--only`参数限制更新范围

## 与相关命令的配合

### 工作流程

```
/编写规格 → /原型设计 → [原型调整] → /更新Spec → /拆分Spec → /设计验证
```

1. **`/编写规格`**：创建初始Spec
2. **`/原型设计`**：基于Spec生成原型
3. **原型调整**：根据反馈调整原型
4. **`/更新Spec`**：根据调整后的原型更新Spec
5. **`/拆分Spec`**：重新拆分更新后的Spec
6. **`/设计验证`**：验证更新后的Spec与原型的一致性

*此命令用于在原型或前端页面发生细微调整后，自动识别变化并更新功能规格文档，确保Spec与实现保持一致。*
