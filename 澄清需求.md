description: 通过提出高度针对性的澄清问题来识别当前功能规格中规格不足的领域，并将答案编码回规格中。问题数量 ≤ 5 个时在对话中一次性展示所有问题，> 5 个时自动生成网页表单。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

目标：检测并减少活动功能规格中的歧义或缺失的决策点，并将澄清直接记录在规格文件中。

注意：此澄清工作流预期在调用 `/制定计划` **之前**运行（并完成）。如果用户明确表示他们正在跳过澄清（例如，探索性尖峰），您可以继续，但必须警告下游返工风险增加。

执行步骤：

1. 从仓库根目录运行 `.specify/scripts/powershell/check-prerequisites.ps1 -Json -PathsOnly` **一次**（组合 `--json --paths-only` 模式 / `-Json -PathsOnly`）。解析最小 JSON 有效载荷字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选择捕获 `IMPL_PLAN`、`TASKS` 以供将来的链式流程。）
   - 如果 JSON 解析失败，则中止并指示用户重新运行 `/编写规格` 或验证功能环境。
   - 对于参数中的单引号如"I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或如果可能使用双引号："I'm Groot"）。

1.5. **保存当前规格文件状态**（⚠️ 关键优化：避免过程中断导致文件无法保存）：
   - 读取当前规格文件内容
   - 创建备份文件：`$FEATURE_DIR/spec-backup-$(date +%Y%m%d-%H%M%S).md`（可选，用于恢复）
   - 确保规格文件已存在且可写
   - 如果规格文件不存在或无法读取，提示用户先运行 `/编写规格`

2. 加载当前规格文件。使用此分类法执行结构化歧义和覆盖扫描。对于每个类别，标记状态：清晰 / 部分 / 缺失。生成用于优先级排序的内部覆盖地图（除非不会提出问题，否则不输出原始地图）。

   功能范围和行为：
   - 核心用户目标和成功标准
   - 明确的范围外声明
   - 用户角色/人物角色区分

   领域和数据模型：
   - 实体、属性、关系
   - 身份和唯一性规则
   - 生命周期/状态转换
   - 数据量/规模假设

   交互和用户体验流程：
   - 关键用户旅程/序列
   - 错误/空/加载状态
   - 无障碍或本地化说明

   非功能质量属性：
   - 性能（延迟、吞吐量目标）
   - 可扩展性（横向/纵向、限制）
   - 可靠性和可用性（正常运行时间、恢复期望）
   - 可观察性（日志记录、指标、跟踪信号）
   - 安全和隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如果有）

   集成和外部依赖：
   - 外部服务/API 和故障模式
   - 数据导入/导出格式
   - 协议/版本假设

   边缘情况和故障处理：
   - 负面场景
   - 速率限制/节流
   - 冲突解决（例如，并发编辑）

   约束和权衡：
   - 技术约束（语言、存储、托管）
   - 明确的权衡或拒绝的替代方案

   术语和一致性：
   - 规范术语表术语
   - 避免的同义词/已弃用的术语

   完成信号：
   - 验收标准可测试性
   - 可衡量的完成定义风格指标

   杂项/占位符：
   - TODO 标记/未解决的决定
   - 缺乏量化的模糊形容词（"健壮"、"直观"）

   对于状态为部分或缺失的每个类别，添加候选问题机会，除非：
   - 澄清不会实质性改变实施或验证策略
   - 信息最好推迟到计划阶段（内部记录）

3. 生成（内部）候选澄清问题的优先级队列。应用这些约束：
    - **重要**：必须生成所有符合条件的候选问题，不要因为担心问题数量而提前停止。
    - **关键数量规则**：
      - 如果检测到需要澄清的类别 ≤ 5 个：生成所有符合条件的候选问题（通常也是 ≤ 5 个），用于对话中一次性展示。
      - 如果检测到需要澄清的类别 > 5 个：**必须生成所有符合条件的候选问题，不设数量上限**。不要限制在 10 个或任何其他数量，让所有需要澄清的重要问题都包含在内。
    - 每个问题必须以以下任一方式回答：
       - 简短的多项选择（2-5 个不同的、互斥的选项），或
       - 一个单词/短语答案（明确约束："用 ≤5 个单词回答"）。
    - 仅包括其答案实质性影响架构、数据建模、任务分解、测试设计、用户体验行为、运营准备或合规验证的问题。
    - 确保类别覆盖平衡：尝试首先覆盖最高影响的未解决类别；当单个高影响领域（例如，安全态势）未解决时，避免提出两个低影响问题。
    - 排除已回答的问题、琐碎的样式偏好或计划级别的执行细节（除非阻止正确性）。
    - 偏爱减少下游返工风险或防止验收测试错位的澄清。
    - 如果超过 5 个类别仍未解决，按（影响 * 不确定性）启发式排序所有问题。
    - **关键**：让步骤 4 根据实际生成的候选问题数量来决定使用对话展示还是网页表单模式。不要因为担心问题太多而提前限制生成的问题数量。

4. **问题数量判断与处理方式选择**：
   - 统计步骤 3 中生成的所有候选澄清问题的数量（记为 N）。
   - **如果 N ≤ 5 个**：继续执行步骤 5 的对话中一次性展示所有问题。
   - **如果 N > 5 个**：执行网页表单生成流程（步骤 4a），跳过对话展示。
   - **注意**：这里的判断基于候选问题队列的实际数量，不要因为担心问题太多而提前限制生成的问题数量。

4a. **网页表单生成流程**（仅在问题数量 > 5 时执行）：
   - 将所有候选问题转换为JSON格式，包含：
     - `title`: 问题标题
     - `context`: 问题上下文（引用相关规格部分）
     - `type`: 问题类型（"choice" 或 "short"）
     - `options`: 选项列表（如果是选择题，包含 `key`, `text`, `meaning`）
     - `recommendation`: 推荐答案（如果适用）
   - 调用脚本生成HTML表单：
     ```bash
     .specify/scripts/bash/generate-clarification-form.sh \
       "$FEATURE_DIR" \
       "$FEATURE_NAME" \
       "$FEATURE_DIR/clarification-questions.json" \
       "$FEATURE_DIR/clarification-form.html"
     ```
   - 将问题数据保存为JSON文件：`$FEATURE_DIR/clarification-questions.json`
   - 生成HTML表单文件：`$FEATURE_DIR/clarification-form.html`
   - 向用户展示：
     ```
     ════════════════════════════════════════════════════
     检测到 [N] 个需要澄清的问题
     ════════════════════════════════════════════════════
     
     为了完整澄清所有需求细节，系统已生成交互式网页表单。
     
     📋 表单文件位置：
     $FEATURE_DIR/clarification-form.html
     
     📝 使用说明：
     1. 在浏览器中打开上述HTML文件
     2. 填写所有问题的答案
     3. 点击"提交答案"按钮
     4. 将下载的JSON文件保存到：$FEATURE_DIR/clarification-answers.json
     5. 完成后，运行以下命令更新规格文档：
     
        /澄清需求 --apply-answers
     
     或者直接告诉我"已填写表单"，我将自动查找并应用答案。
     
     ════════════════════════════════════════════════════
     ```
   - 等待用户完成表单填写并提交。
   - 检测到 `$FEATURE_DIR/clarification-answers.json` 文件存在后，执行步骤 4b。

4b. **应用网页表单答案**（当检测到答案文件时，⚠️ 关键优化：逐个处理答案并立即更新文件）：
   - 读取 `$FEATURE_DIR/clarification-answers.json` 文件
   - 解析JSON文件，获取所有答案
   - **逐个处理每个答案并立即更新文件**：
     - 对于每个答案：
       1. 将答案记录到澄清部分：`- 问：<问题> → 答：<最终答案>`。
       2. 根据答案内容，智能更新Spec的相应部分（功能需求、用户故事、数据模型等）
       3. **立即更新文件**：将当前答案的澄清内容写入规格文件
   - 所有答案处理完成后，继续执行步骤 6（验证）和后续步骤

5. **对话中一次性展示所有问题**（仅在问题数量 ≤ 5 时执行）：
   - **重要**：一次性展示所有问题（1-5个），而不是递进式一题一题地问。
   - 将所有问题按顺序展示，每个问题包含：
     - 问题标题和上下文
     - 推荐答案（如果有）
     - 所有选项（如果是选择题）或简短答案提示
   - 格式示例：
     ```
     ════════════════════════════════════════════════════
     检测到 [N] 个需要澄清的问题
     请一次性回答所有问题
     ════════════════════════════════════════════════════
     
     **问题 1：[问题标题]**
     上下文：[问题上下文，引用相关规格部分]
     
     推荐：选项 [X] - <理由>
     
     选项：
     - A: <选项A描述>
     - B: <选项B描述>
     - C: <选项C描述>
     - 自定义: 提供您自己的答案（≤5个单词）
     
     **问题 2：[问题标题]**
     ...
     
     **问题 N：[问题标题]**
     ...
     
     ════════════════════════════════════════════════════
     请按以下格式一次性提交所有答案：
     
     问题1: [您的答案]
     问题2: [您的答案]
     ...
     问题N: [您的答案]
     
     或者：
     - 如果接受所有推荐答案，回复"接受所有推荐"
     - 如果部分接受推荐，按格式提交，未指定的问题使用推荐答案
     ═════════════════════════════════════════════════════
     ```
   - 用户提交答案后：
     - 解析所有答案
     - 如果用户回复"接受所有推荐"，使用所有推荐答案
     - 如果部分问题未回答，使用推荐答案（如果有）
     - 验证所有答案的格式和完整性
     - 将所有答案记录到工作内存，然后执行步骤 5（集成答案）
   - **不再使用递进式提问**：不要一题一题地问，必须一次性展示所有问题。

5b. **版本管理**（在集成答案之前执行，⚠️ 必须执行）：
    - **必须先加载并完全遵循**模块文件：`.cursor/modules/spec-version-management.md`
    - 读取当前版本号（从 `FEATURE_DIR/VERSION` 文件，如果不存在则默认为 v1.0.0）
    - 计算新版本号（Patch版本递增，如：v1.0.0 → v1.0.1）
    - 保存旧版本到 `FEATURE_DIR/versions/{旧版本号}-spec.md`
    - 更新 `FEATURE_DIR/VERSION` 文件为新版本号
    - 准备CHANGELOG记录：
      - 变更类型：🔍 澄清需求
      - 变更原因：基于澄清的问题总结
      - 变更内容：列出所有澄清的答案要点
      - 影响范围：说明澄清对开发的影响
      - 验证状态：✅ 已通过澄清需求验证

5c. **集成所有答案**（在收到所有答案后执行，⚠️ 关键优化：每处理一个答案立即更新文件）：
    - 维护规格的内存表示（在开始时加载一次）加上原始文件内容。
    - 对于本会话中第一个集成的答案：
       - 确保存在 `## 澄清` 部分（如果缺少，则根据规格模板在最高级别的上下文/概述部分之后创建它）。
       - 在其下，为今天创建（如果不存在）`### 会话 YYYY-MM-DD` 子标题。
       - **立即更新文件**：将澄清部分结构写入规格文件
       - **更新版本信息**：在spec.md头部更新版本号，在版本信息章节添加新版本记录
    - **逐个处理答案并立即更新文件**（避免一次性处理导致中断时丢失所有内容）：
       - 对于每个答案：
         1. 将答案附加到澄清部分：`- 问：<问题> → 答：<最终答案>`。
         2. 将澄清应用于最合适的部分：
            - 功能歧义 → 更新或添加功能需求中的项目符号。
            - 用户交互/参与者区分 → 更新用户故事或参与者子部分（如果存在）与澄清的角色、约束或场景。
            - 数据形状/实体 → 更新数据模型（添加字段、类型、关系）保留排序；简洁地记录添加的约束。
            - 非功能约束 → 在非功能/质量属性部分添加/修改可衡量的标准（将模糊形容词转换为指标或明确目标）。
            - 边缘情况/负面流程 → 在边缘情况/错误处理下添加新项目符号（或如果模板为其提供占位符，则创建此类子部分）。
            - 术语冲突 → 在规格中规范化术语；仅在必要时通过添加一次 `（以前称为"X"）` 来保留原始术语。
         3. **立即更新文件**：将当前答案的澄清内容写入规格文件，确保即使后续处理中断，已处理的答案也已保存。
    - 如果澄清使较早的模糊陈述无效，则替换该陈述而不是重复；不留下过时的矛盾文本。
    - 保留格式：不要重新排序不相关的部分；保持标题层次结构完整。
    - 保持每个插入的澄清最小且可测试（避免叙述漂移）。
    - **最终保存**：在所有答案处理完成后，再次保存规格文件以确保完整性。

6. 验证（在应用所有答案后执行）：
   - 澄清会话每个接受的答案恰好包含一个项目符号（没有重复）。
   - 所有问题都已回答（对话模式：≤5个问题；网页表单模式：所有问题）。
   - 更新的部分不包含新答案旨在解决的挥之不去的模糊占位符。
   - 没有矛盾的较早陈述保留（扫描现在无效的替代选择已删除）。
   - Markdown 结构有效；仅允许新标题：`## 澄清`、`### 会话 YYYY-MM-DD`。
   - 术语一致性：在所有更新的部分中使用相同的规范术语。

7. 将更新的规格写回 `FEATURE_SPEC`。

8. **更新CHANGELOG.md**（在更新Spec后执行）：
   - 在 `FEATURE_DIR/CHANGELOG.md` 文件开头（`---` 之后）添加新版本记录
   - 包含完整的变更信息（变更类型、原因、内容、影响范围、验证状态）

9. 报告完成（在提问循环结束或提前终止后）：
   - 提出和回答的问题数量。
   - 更新的规格路径。
   - 触及的部分（列出名称）。
   - 覆盖摘要表，列出每个分类类别，状态：已解决（以前是部分/缺失并已解决）、推迟（超过问题配额或更适合规划）、清晰（已经足够）、未解决（仍然是部分/缺失但影响低）。
   - 如果仍有未解决或推迟的问题，建议是继续 `/制定计划` 还是在计划后再次运行 `/澄清需求`。
   - 建议的下一个命令。

行为规则：

- 如果未发现有意义的歧义（或所有潜在问题都将是低影响的），回复："未检测到值得正式澄清的关键歧义。"并建议继续。
- 如果规格文件缺失，指示用户首先运行 `/编写规格`（不要在此处创建新规格）。
- **问题数量处理**：
  - **关键**：在步骤 3 中，必须生成所有符合条件的候选问题，不要因为担心问题数量而提前停止生成。
    - 如果检测到需要澄清的类别 ≤ 5 个：生成所有符合条件的候选问题（通常也是 ≤ 5 个）。
    - 如果检测到需要澄清的类别 > 5 个：**必须生成所有符合条件的候选问题，不设数量上限**。
  - **处理方式选择**：
    - 如果候选问题数量 ≤ 5 个：在对话中一次性展示所有问题（步骤 5），让用户一次性提交所有答案
    - 如果候选问题数量 > 5 个：自动生成网页表单（步骤 4a），不进行对话展示
  - **不再使用递进式提问**：5个及以内的问题也要一次性展示，不要一题一题地问。
- 避免推测性技术栈问题，除非缺少会阻止功能清晰度。
- 尊重用户的提前终止信号（"停止"、"完成"、"继续"）。
- 如果由于完全覆盖而未提出问题，则输出紧凑的覆盖摘要（所有类别都清晰），然后建议前进。
- 如果配额达到，仍有未解决的高影响类别，请在推迟下明确标记它们并提供理由。
- **对话展示模式**（≤5个问题）：在Cursor对话中一次性展示所有问题，让用户一次性提交所有答案，不再使用递进式一题一题的方式。
- **网页表单模式**（>5个问题）：当问题数量超过5个时，使用网页表单模式，所有问题都会包含在表单中，用户可以一次性查看所有问题并填写所有答案。

优先级的上下文：$ARGUMENTS
