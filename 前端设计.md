## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于根据功能规格文档或用户需求直接生成前端页面代码。

**核心功能**：
1. 支持指定设计规范（如Kubeease、Common-Admin等）
2. 如果未指定设计规范，自动使用最佳实践的设计规范
3. 根据规格文档或用户需求直接生成前端页面代码

**重要说明**：
- 本命令直接生成**前端页面代码**（Vue/React组件）
- 设计遵循指定的设计规范或最佳实践规范
- 不生成设计文档、设计说明文档或设计规范引用文档

## 执行流程

### 1. 前置检查与输入识别

#### 1.1 检查输入源

根据用户输入识别设计需求来源：

- **规格文档路径**：用户指定 `--spec=<path>` 或输入包含 `.md` 文件路径
- **功能描述**：用户直接描述功能需求（如"用户管理列表页"）
- **现有页面**：用户指定现有页面路径，基于现有页面进行设计优化

#### 1.2 确定基础工程路径

**基础工程确定规则**（按优先级）：

1. **如果用户指定了 `--base-project=<path>` 参数**：
   - 使用指定的基础工程路径

2. **如果当前工作目录在某个前端项目中**：
   - 使用当前目录作为基础工程

3. **如果设置了基础工程配置**（通过 `/初始工程下载` 设置）：
   - 使用配置中的工程路径

4. **如果 Spec 目录下有 `initial-project/`**：
   - 提示用户：是否基于 `docs/specs/{编号}-{名称}/initial-project/` 生成？
   - 如果用户确认，使用该路径

5. **默认情况**：
   - 使用 `rules/company/Product-Design/rp-generate-v3-back`（Common-Admin）

#### 1.3 检查是否为初始工程并处理菜单

**检查是否为初始工程**：
- 检查基础工程路径是否以 `-initial` 结尾
- 或者检查工程目录名称是否匹配初始工程命名模式（如 `docs/frontend-projects/{项目名}-initial`）

**如果检测到是初始工程**：

1. **自动隐藏初始工程菜单**：
   - 查找初始工程的路由配置文件（`src/router/index.ts`、`src/router/const.ts` 或类似文件）
   - 给初始工程的所有业务路由添加 `meta: { hidden: true }` 标记
   - 确保 `transformRoutesToMenus` 函数能正确过滤隐藏的路由
   - 仅基于Spec内容生成菜单
   - 提示用户：已自动隐藏初始工程菜单，菜单将仅基于Spec内容生成
   - 记录菜单处理方式：`hide`

2. **记录菜单处理方式**：
   - 在对话上下文中记录菜单处理方式为 `hide` 和基础工程路径
   - 后续生成菜单时使用此配置

**如果检测到不是初始工程**：
- 跳过菜单处理步骤，直接进入下一步
- 记录菜单处理方式：`none`

#### 1.4 配置路径设置（仅针对初始工程）

**⚠️ 重要**：此步骤仅在检测到是初始工程时执行，确保前端设计后的项目能正常预览，不会因为路径问题导致页面报错或图片不显示。

**如果检测到是初始工程**：

1. **检查并配置 `vite.config.ts`**：
   
   - 读取 `vite.config.ts` 文件
   - 检查 `base` 配置：
     - 如果 `base` 是固定值（如 `'./'` 或 `'/'`）→ 需要修改为动态配置
     - 如果 `base` 已经是动态配置（根据 `mode` 判断）→ 跳过修改
   
   - **修改方式**：
     - 在 `defineConfig` 函数开始处添加：
       ```typescript
       // 根据模式设置不同的 base 路径
       // 开发模式：使用绝对路径 '/'，确保开发服务器正常工作
       // 生产模式：使用相对路径 './'，确保ACW预览能正确加载资源
       const base = mode === 'production' ? './' : '/'
       ```
     - 将 `return` 对象中的 `base` 配置改为使用变量：`base,`
     - 添加注释说明配置原因
   
   - **配置示例**：
     ```typescript
     export default defineConfig(({ mode }) => {
       const env = loadEnv(mode, process.cwd(), '')
       const appCode = env.VITE_APP_CODE || 'vpp-platform'
       
       // 根据模式设置不同的 base 路径
       // 开发模式：使用绝对路径 '/'，确保开发服务器正常工作
       // 生产模式：使用相对路径 './'，确保ACW预览能正确加载资源
       const base = mode === 'production' ? './' : '/'
       
       return {
         base, // 动态设置 base 路径
         // ... 其他配置
       }
     })
     ```

2. **检查并配置 `index.html`**：
   
   - 读取 `index.html` 文件
   - 检查 favicon 路径：
     - 如果是绝对路径（如 `href="/favicon.ico"`）→ 修改为相对路径 `href="./favicon.ico"`
     - 如果已经是相对路径 → 跳过修改
   
   - **修改方式**：
     - 查找 `<link rel="icon"` 标签
     - 将 `href="/favicon.ico"` 改为 `href="./favicon.ico"`
     - 添加注释说明（可选）

3. **验证图片资源导入方式**：
   
   - 检查关键图片文件（如 logo）是否使用 `import` 方式导入
   - 检查 `src/layouts/header/AdminHeader.vue` 和 `src/layouts/AdminLayout.vue` 中的图片导入
   - 如果使用 `import` 导入（如 `import logo from '@/assets/images/logo.png'`）→ ✅ 正确，无需修改
   - 如果使用字符串路径（如 `src="/assets/logo.png"`）→ ⚠️ 建议改为 `import` 方式

4. **配置完成提示**：
   ```
   ✅ 已自动配置路径设置（仅针对初始工程）
   
   配置内容：
   - vite.config.ts：已配置动态 base 路径
     * 开发模式：base = '/'（绝对路径）
     * 生产模式：base = './'（相对路径，用于ACW预览）
   - index.html：favicon 已设置为相对路径
   - 图片资源：使用 import 导入，路径自动处理
   
   现在可以确保：
   - 开发环境正常访问，图片正常显示
   - 生产构建后ACW预览正常，资源路径正确
   - 不会因为路径问题导致页面报错或图片不显示
   ```

**如果检测到不是初始工程**：
- 跳过此步骤，不进行任何路径配置修改
- 保持原有配置不变

**注意事项**：
- 此配置仅在前端设计时执行，不影响初始工程下载后的原始状态
- 初始工程下载后，工程保持原样，可以正常打开默认首页
- 只有在执行前端设计命令时，才会自动配置路径设置

#### 1.5 读取规格文档（如果指定）

如果用户指定了规格文档：
- 读取规格文档内容
- 提取关键信息：功能名称、用户故事、功能需求、数据模型、页面类型、交互需求
- **提取菜单信息**：从Spec中提取菜单结构、菜单项、菜单层级、菜单图标等信息

#### 1.6 确定设计范围

- **页面类型**：列表页、表单页、详情页、仪表板等
- **功能模块**：用户管理、订单管理、数据统计等
- **设计范围**：单个页面、完整模块、多个页面

### 2. 设计规范识别与加载

#### 2.1 用户指定设计规范

如果用户指定了设计规范（通过 `--design-system=<name>` 参数）：

**支持的设计规范**：
- `kubeease` - Kubeease平台设计规范
  - 规范路径：`rules/company/Product-Design/kubeease-design-specs/`
  - 适用场景：Kubeease平台相关项目
- `common-admin` - Common-Admin后管平台设计规范
  - 规范路径：`rules/company/Product-Design/rp-generate-v3-back/docs/`
  - 适用场景：后管平台项目
- `mobile` - 移动端设计规范
  - 规范路径：`rules/company/Product-Design/mobile-design-rules.md`
  - 适用场景：移动端应用
- `custom` - 自定义设计规范
  - 用户指定自定义规范路径

**加载规范文件**：
- `design-system.md` - 核心设计系统（颜色、间距、字体、布局）
- `component-specs.md` - 组件详细规范（按钮、表单、表格等）
- `design-tokens.css` / `design-tokens.less` - 设计变量
- `prototype-guide.md` - 原型设计指南（如适用）
- `frontend-guide.md` - 前端开发指南（如适用）

#### 2.2 自动识别设计规范（未指定时）

如果用户未指定设计规范，根据以下规则自动识别：

1. **项目类型识别**：
   - 检查项目目录结构、配置文件、现有代码
   - 识别项目类型：Kubeease项目、Common-Admin项目、移动端项目、通用项目

2. **最佳实践规范选择**：
   - **后管平台项目** → 使用 Common-Admin 设计规范
   - **Kubeease项目** → 使用 Kubeease 设计规范
   - **移动端项目** → 使用移动端设计规范
   - **通用项目** → 使用通用最佳实践设计规范

3. **通用最佳实践设计规范**（当无法识别项目类型时）：
   - 基于行业最佳实践
   - 主色：#1677ff（蓝色）
   - 布局：24栅格系统，内容区宽度1200px
   - 组件：按钮高度32px，表格行高48px，卡片圆角8px

#### 2.3 确认设计规范

向用户显示：
- 识别的设计规范名称
- 规范文件路径
- 核心设计参数（颜色、间距、组件尺寸等）
- 询问用户是否确认或需要修改

### 3. 页面场景识别

根据规格文档或用户需求，识别页面类型（参考 `rules/company/Product-Design/page-scenario-library.md`）：

**页面类型映射**：
- 列表、管理、查询 → 列表分页页（S11）
- 新增、创建、添加 → 新增表单页（S12）
- 编辑、修改、更新 → 编辑表单页（S12）
- 详情、查看 → 查看详情页（S13）
- 仪表板、统计、概览 → 数据概览页（S20）
- 搜索、筛选 → 高级搜索页（S15）

### 4. 生成前端页面代码

#### 4.1 确定输出位置

根据项目结构确定前端页面代码的输出位置：
- 检查项目中的 `src/` 目录结构
- 确定页面组件应该放置的目录（如 `src/pages/`、`src/views/` 等）
- 根据模块创建相应的目录结构

#### 4.2 生成页面组件代码

直接生成前端页面组件代码（Vue/React），包括：

**1. 页面结构**
- 导入必要的依赖（组件库、工具函数等）
- 定义组件的基本结构
- 设置组件的 props 和 state

**2. 布局实现**
- 使用设计规范中的布局系统
- 实现响应式布局
- 应用栅格系统

**3. 组件使用**
- 使用组件库中的标准组件（按钮、表单、表格等）
- 配置组件的属性（尺寸、颜色、样式等）
- 实现组件的交互逻辑

**4. 数据交互**
- 实现数据获取逻辑（API调用）
- 实现数据展示（表格、列表等）
- 实现数据操作（新增、编辑、删除等）

**5. 样式应用**
- 使用设计规范中的设计变量（design tokens）
- 应用颜色、间距、字体等样式
- 确保样式符合设计规范

#### 4.3 生成路由和菜单配置

**基于Spec内容生成路由配置**：

1. **从Spec中提取菜单结构信息**：
   - 提取功能模块、页面层级、菜单项名称、菜单图标等
   - 根据Spec中的模块划分确定菜单结构

2. **为每个页面生成路由配置**：
   - 为每个生成的页面组件创建对应的路由配置
   - 设置路由的 `path`、`name`、`component` 等属性
   - 设置路由的 `meta` 信息（title、icon、hidden等）

3. **自动隐藏初始工程菜单**（如果检测到是初始工程）：
   
   - 检查基础工程的路由配置文件
   - 给所有初始工程的业务路由添加 `meta: { hidden: true }` 标记
   - 添加注释说明：`// 初始工程菜单已自动隐藏，仅显示基于Spec生成的菜单`
   - 确保新生成的路由不包含 `hidden: true` 标记

4. **生成或更新路由文件**：
   - 更新或创建 `src/router/const.ts` 或 `src/router/index.ts`
   - 添加基于Spec的路由配置
   - 确保路由配置包含正确的 `meta` 信息（title、icon、hidden等）
   - 如果使用了 `businessRoutes` 数组，将新路由添加到该数组

5. **菜单配置说明**：
   - 确保 `transformRoutesToMenus` 函数能正确过滤 `hidden: true` 的路由
   - 菜单将自动从路由配置中生成
   - 初始工程菜单已自动隐藏，菜单将仅显示基于Spec生成的路由

6. **智能更新默认首页**：
   
   **触发条件检查**：
   - 读取 `src/router/index.ts` 文件
   - 检查当前默认路由（根路径 `/` 的 `redirect` 值）
   - 如果默认路由是 `/home`（占位页）→ 执行更新
   - 如果默认路由已经是业务页面 → 跳过更新
   
   **确定新的默认首页**（按优先级）：
   
   a. **优先：从Spec文档中提取默认首页**：
      - 检查Spec文档中是否明确标记了"首页"、"默认页面"、"首页路径"等信息
      - 如果找到，提取首页路径（如 `/overview/index`）
      - 检查对应的页面文件是否存在（如 `src/views/overview/index.vue`）
      - 如果文件存在 → 使用该路径作为新的默认首页
   
   b. **其次：使用第一个基于Spec生成的菜单页面**：
      - 从基于Spec生成的路由配置中，找到第一个业务路由
      - 获取该路由的第一个子路由（或路由本身如果没有子路由）
      - 构建完整路径（如 `/overview/index`）
      - 检查对应的页面文件是否存在
      - 如果文件存在 → 使用该路径作为新的默认首页
      - 如果文件不存在 → 跳过更新，提示用户
   
   c. **如果以上都不满足**：
      - 保持原样（指向 `/home`），提示用户
   
   **执行更新**：
   
   1. **更新路由配置文件**（`src/router/index.ts`）：
      - 查找 `redirect: '/home'` 或 `redirect: '/'`
      - 替换为 `redirect: '{新的默认首页路径}'`
      - 添加注释说明：`// 默认首页已更新为基于Spec设计的首页`
   
   2. **更新登录跳转**（`src/views/login/index.vue`）：
      - 查找登录成功后的跳转逻辑
      - 查找 `const path = process.env.NODE_ENV === 'development' ? '/home' : '/'` 或类似代码
      - 替换为 `const path = '{新的默认首页路径}'`
      - 或者统一改为：`const path = '{新的默认首页路径}'`
   
   **更新提示**：
   ```
   ✅ 已智能更新默认首页
   
   原默认路由：/home（占位页）
   新默认首页：{新的默认首页路径}
   
   现在访问根路径将自动跳转到基于Spec设计的首页。
   ```
   
   **如果无法更新**（页面文件不存在或没有业务路由）：
   ```
   ⚠️ 未更新默认首页
   
   原因：{原因说明}
   当前默认路由：/home（占位页）
   
   建议：在生成第一个业务页面后，默认首页会自动更新。
   ```

7. **确保默认首页配置正确**：
   
   - 确保根路径 `/` 的 `redirect` 指向业务首页（如 `/overview/index`）
   - 确保登录成功后跳转到业务首页
   - 确保 `index.html` 是应用的入口文件
   - 如果路径配置已正确（在步骤1.4中已配置），则无需额外处理
   - 确保开发模式和生产模式都能正常访问首页

#### 4.4 代码质量保证

- 遵循项目的代码规范
- 添加必要的注释
- 确保代码可读性和可维护性
- 处理边界情况和错误处理

### 7. 浏览器验证与错误修复

**执行模块**：`.cursor/modules/错误检测与监控.md`

设计完成后，自动启动本地服务器，使用Chrome浏览器打开页面进行设计验证，确保页面没有错误。

**步骤**：
1. 启动开发服务器（优先：`npm run dev` / `yarn dev` / `pnpm dev`，备选：`npm run serve` / `npm start`）
2. 使用Chrome浏览器打开页面（优先Chrome，备选系统默认浏览器）
3. 执行错误检测与自动修复（详见模块文档）
4. 生成验证报告

**成功标准**：
- ✅ 服务器成功启动
- ✅ 页面成功加载（HTTP 200）
- ✅ 无编译错误
- ✅ 无运行时错误
- ✅ 页面正常渲染

**终止条件**：
- ✅ 页面成功加载，无错误 → **成功，保持运行**
- ❌ 无法自动修复的错误 → **暂停，显示错误详情，等待用户修复**
- ⏹️ 用户手动终止（Ctrl+C）→ **停止服务器**

## 命令参数说明

```
/前端设计 [选项] [功能描述]

选项:
  --spec=<path>             指定功能规格文档路径
  --base-project=<path>     指定基础工程路径（如果未指定，会自动识别）
  --design-system=<name>     指定设计规范（kubeease/common-admin/mobile/custom）
  --design-system-path=<path> 自定义设计规范路径（当--design-system=custom时）
  --page-type=<type>        指定页面类型（list/form/detail/dashboard）
  --output=<path>           指定页面组件输出路径
  --no-verify               跳过浏览器验证步骤（仅生成代码，不启动服务器和浏览器）
  --no-auto-fix             禁用自动修复（仅检测错误，不自动修复）
  --browser=<browser>       指定浏览器（chrome/firefox/safari/default，默认：chrome）
  --port=<port>             指定开发服务器端口（默认：自动选择）

功能描述:
  如果未指定--spec，可以直接描述功能需求

示例:
  /前端设计 --spec=docs/specs/user/spec.md --design-system=kubeease
  /前端设计 用户管理列表页 --design-system=common-admin
  /前端设计 --spec=docs/specs/order/spec.md  # 自动识别设计规范
  /前端设计 --spec=docs/specs/user/spec.md --base-project=docs/frontend-projects/my-project-initial  # 指定基础工程
  /前端设计 --spec=docs/specs/user/spec.md --no-verify  # 仅生成代码，不验证
  /前端设计 --spec=docs/specs/user/spec.md --browser=chrome --port=5173  # 指定浏览器和端口
```

## 设计规范说明

支持的设计规范：
- `kubeease` - Kubeease平台设计规范（主色：#04a7b2）
- `common-admin` - Common-Admin后管平台设计规范（主色：#1677ff）
- `mobile` - 移动端设计规范
- `custom` - 自定义设计规范

未指定时自动识别项目类型并选择对应规范，无法识别时使用通用最佳实践规范。

## 注意事项

1. **设计规范优先级**：用户指定 > 项目类型识别 > 最佳实践规范

2. **代码输出**：默认保存在 `src/pages/` 或 `src/views/`，可通过 `--output` 自定义

3. **浏览器验证**：默认启用，使用 `--no-verify` 跳过；自动检测并修复错误直到解决

4. **与原型设计的区别**：本命令生成前端页面代码（Vue/React组件），`/原型设计` 生成HTML原型页面

5. **初始工程菜单处理**：
   - 如果检测到是基于初始工程下载的项目（路径包含 `-initial`），会自动隐藏初始工程的示例菜单
   - 菜单将仅显示基于Spec生成的菜单
   - 菜单配置会自动更新路由文件，确保菜单仅基于Spec内容生成
   - 默认首页会自动更新为基于Spec设计的首页（如果Spec中标记了首页）或第一个菜单页面

6. **初始工程路径配置**：
   - 如果检测到是基于初始工程下载的项目，会自动配置路径设置
   - 开发模式使用绝对路径（`base: '/'`），确保开发服务器正常工作
   - 生产模式使用相对路径（`base: './'`），确保ACW预览能正确加载资源
   - 自动配置 favicon 为相对路径，确保图片资源正常显示
   - 此配置仅在前端设计时执行，不影响初始工程下载后的原始状态
