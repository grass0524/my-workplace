## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

本命令用于在功能规格文档（Spec）完成后，生成标准的产品需求文档（PRD）。PRD 文档用于需求评审、产品规划和开发指导。

**重要说明**：
- 本命令基于功能规格文档（Spec）生成 PRD
- PRD 模板遵循 `.specify/templates/prd-template.md` 规范
- PRD 文档应保存在 `docs/prd/` 目录下，命名格式：`PRD-{模块名称}.md`
- PRD 与 Spec 的关系：Spec 侧重 What & Why，PRD 侧重 How（业务层面）

## 执行流程

### 1. 前置检查

a. **检查功能规格文档**：
   - 如果用户指定了规格文档路径，使用指定路径
   - 如果未指定，搜索项目中的 `spec.md` 文件（优先查找 `docs/specs/` 目录）
   - 如果找到多个规格文档，列出所有文档让用户选择
   - 如果找不到规格文档，提示用户先运行 `/编写规格` 命令

b. **读取规格文档**：
   - 读取规格文档内容
   - 提取关键信息：
     - 功能名称、模块名称
     - 用户故事、功能需求
     - 数据模型、字段定义
     - 页面类型、交互流程
     - 成功标准、验收场景

c. **确定输出路径**：
   - 如果规格文档在 `docs/specs/{编号}-{模块名}/spec.md`
     - PRD 输出路径：`docs/prd/PRD-{模块名}.md`
   - 如果规格文档在其他位置
     - PRD 输出路径：与 spec.md 同目录下的 `PRD-{模块名}.md`
   - 如果 `docs/prd/` 目录不存在，自动创建

### 2. 加载 PRD 模板并创建初始文件

a. **读取模板文件**：
   - 读取 `.specify/templates/prd-template.md` 模板
   - 理解模板结构和内容要求

b. **理解模板要求**：
   - PRD 通用结构（11个主要章节）
   - 功能架构图要求（推荐使用 PlantUML）
   - 模块调用关系图要求（推荐使用 PlantUML）
   - 流程图要求（推荐使用 PlantUML 或 Mermaid）
   - 字段定义表格格式
   - 状态说明与流转图要求

c. **立即创建并初始化PRD文件**（⚠️ 关键优化：避免过程中断导致文件无法保存）：
   - 根据步骤1.c确定的输出路径创建PRD文件
   - 创建初始PRD文件，包含：
     * 文档头部（PRD名称、版本号、创建时间、基于的Spec文档链接）
     * 基本章节结构（11个主要章节的标题和占位符）
     * 每个章节使用 `[待填充]` 标记，表示内容将在后续步骤中填充
   - **立即写入PRD文件**，确保文件已创建
   - 这样即使后续步骤中断，文件也已经存在并包含基本结构

### 3. 从 Spec 提取信息并映射到 PRD（⚠️ 关键优化：每完成一个章节立即更新文件）

根据 PRD 模板结构，从 Spec 中提取并转换信息：

**3.1 背景与目标**：
- 从 Spec 的"背景"或"概述"提取业务背景
- 从 Spec 的"目标"或"成功标准"提取核心目标
- 从 Spec 的"成功标准"提取成功指标
- **立即更新文件**：将背景与目标内容写入PRD文件的"一、背景与目标"章节，替换 `[待填充]` 标记

**3.2 用户与角色**：
- 从 Spec 的"用户故事"提取目标用户
- 从 Spec 的"用户角色"或"权限"提取用户角色
- 从 Spec 的"使用场景"提取使用场景
- 从 Spec 的"权限定义"提取权限定义
- **立即更新文件**：将用户与角色内容写入PRD文件的"二、用户与角色"章节，替换 `[待填充]` 标记

**3.3 系统功能架构**（⭐ 必填）：
- 根据 Spec 的功能模块，绘制功能架构图
- **推荐使用 PlantUML**（支持长线样式、颜色分组、注释说明）
- 展示系统分层结构（前端层、业务层、数据层等）
- 展示功能模块划分和模块间的层次关系
- 生成功能模块说明表格
- **立即更新文件**：将功能架构图和说明写入PRD文件的"三、系统功能架构"章节，替换 `[待填充]` 标记

**3.4 概念与数据结构**：
- 从 Spec 的"关键实体"或"数据模型"提取核心概念
- 从 Spec 的"字段定义"提取数据模型
- 生成字段说明表格（包含类型、校验规则、默认值）
- 绘制关系图（如适用）
- **立即更新文件**：将概念与数据结构内容写入PRD文件的"四、概念与数据结构"章节，替换 `[待填充]` 标记

**3.5 功能设计**（逐模块）：
- 从 Spec 的"功能需求"提取每个模块的功能目标
- 从 Spec 的"字段定义"提取字段与规则
- 从 Spec 的"用户故事"和"验收场景"提取主流程
- 从 Spec 的"边界条件"提取异常流程处理
- 使用 Mermaid Flowchart 或 PlantUML 绘制主流程图
- 标注操作风险分级（🟢低风险/🟡中风险/🔴高风险/⚫最高风险）
- **立即更新文件**：将功能设计内容写入PRD文件的"五、功能设计"章节，替换 `[待填充]` 标记

**3.6 交互说明**：
- 从 Spec 的"页面类型"推断页面结构
- 从 Spec 的"功能需求"提取组件行为
- 从 Spec 的"交互流程"提取交互反馈
- 从 Spec 的"验证规则"提取错误提示
- **立即更新文件**：将交互说明内容写入PRD文件的"六、交互说明"章节，替换 `[待填充]` 标记

**3.7 状态说明与流转**：
- 从 Spec 的"状态定义"提取状态说明
- 生成状态说明表格
- 使用 PlantUML 或 Mermaid 绘制状态流转流程图
- 如果有多维度的状态，需要区分不同维度的状态，并给与不同的状态说明和流程图
- 标注操作风险分级
- **立即更新文件**：将状态说明与流转内容写入PRD文件的"七、状态说明与流转"章节，替换 `[待填充]` 标记

**3.8 接口与事件**（如有）：
- 从 Spec 的"API接口"提取接口定义
- 从 Spec 的"事件触发"提取事件机制
- 绘制模块调用关系图（⭐ 推荐使用 PlantUML，支持不同线型区分调用方式）
- 生成模块调用关系说明表格
- 说明数据流向
- **立即更新文件**：将接口与事件内容写入PRD文件的"八、接口与事件"章节，替换 `[待填充]` 标记

**3.9 非功能性需求**：
- 从 Spec 的"性能要求"提取性能要求
- 从 Spec 的"可访问性"提取可访问性要求
- 从 Spec 的"可靠性"提取可靠性要求
- 从 Spec 的"安全性"提取安全性要求
- **立即更新文件**：将非功能性需求内容写入PRD文件的"九、非功能性需求"章节，替换 `[待填充]` 标记

**3.10 依赖与风险**：
- 从 Spec 的"技术依赖"提取技术依赖
- 从 Spec 的"外部依赖"提取外部依赖
- 从 Spec 的"潜在风险"提取潜在风险
- 从 Spec 的"应对方案"提取应对方案
- **立即更新文件**：将依赖与风险内容写入PRD文件的"十、依赖与风险"章节，替换 `[待填充]` 标记

**3.11 变更记录**：
- 初始化版本号为 v1.0
- 记录创建日期
- 记录变更内容（初始创建）
- **立即更新文件**：将变更记录内容写入PRD文件的"十一、变更记录"章节，替换 `[待填充]` 标记

**重要**：每个章节完成后立即更新文件，确保即使过程中断，已完成的内容也已保存。

### 4. **最终文档整理与验证**（⚠️ 注意：文件已在步骤2.c和步骤3中逐步更新，此步骤进行最终整理）

a. **文档结构完整性检查**（确保所有部分都已填充）：
   - 检查所有11个主要章节是否都已填充内容
   - 确保所有 `[待填充]` 标记已被替换为实际内容
   - 验证章节顺序符合模板要求

b. **图表完整性检查**：
   - **功能架构图**：⭐ 推荐使用 PlantUML（支持长线样式、颜色分组）- 应在步骤3.3中完成
   - **模块调用关系图**：⭐ 推荐使用 PlantUML（支持不同线型区分调用方式）- 应在步骤3.8中完成
   - **流程图**：推荐使用 PlantUML 或 Mermaid - 应在步骤3.5中完成
   - **状态图**：推荐使用 Mermaid State Diagram - 应在步骤3.7中完成
   - 确保所有图表都已包含在 PRD 文档中

c. **字段定义标准化验证**：
   - 使用标准表格格式
   - 包含字段名、类型、必填、说明、校验规则、默认值
   - 使用标准验证规则（手机号、邮箱、密码等）
   - 确保所有字段定义表格格式统一

d. **操作风险分级标注验证**：
   - 根据 `docs/rules/prototyping/page-design-rules.md` 的操作分级规范
   - 确保在 PRD 中所有操作都已标注风险等级
   - 确保确认机制和提示文案说明完整

e. **最终保存**：确保所有内容都已写入PRD文件

### 5. 质量检查

根据 PRD 质量标准（模板中的"七、PRD 质量标准"）检查生成的 PRD：

a. **结构完整性**：
   - [ ] 包含所有必备章节（背景、目标、用户、功能设计等）
   - [ ] 每个模块都有完整的内容要求
   - [ ] 变更记录完整可追溯

b. **逻辑清晰性**：
   - [ ] 流程图完整准确（推荐使用 PlantUML）
   - [ ] 功能架构图清晰展示系统分层和模块划分（⭐ 推荐使用 PlantUML）
   - [ ] 模块调用关系图准确展示模块间的调用关系（⭐ 推荐使用 PlantUML）
   - [ ] 业务逻辑描述清晰无歧义
   - [ ] 异常流程和边界条件说明完整

c. **字段规范性**：
   - [ ] 所有字段都有类型、校验规则和默认值
   - [ ] 使用标准的验证规则（引用通用规则库）
   - [ ] 字段说明详细准确

d. **边界清晰性**：
   - [ ] 明确说明本期包含和不包含的内容
   - [ ] 标注未决问题和需要讨论的点
   - [ ] 功能范围界定清晰

e. **可执行性**：
   - [ ] 开发人员能够基于 PRD 直接开发
   - [ ] 测试人员能够基于 PRD 编写测试用例
   - [ ] 产品经理能够基于 PRD 进行需求验收

### 6. **更新索引**（如适用）

a. **更新索引文件**：
   - 如果存在 `docs/prd/README.md`，更新索引
   - 添加新生成的 PRD 链接
   - **注意**：PRD文件已在步骤2.c和步骤3中逐步更新并保存，此步骤仅更新索引

### 7. 报告完成

报告完成，包括：
- 功能名称、模块名称
- PRD 文件路径
- 质量检查结果
- 下一阶段的建议（`/原型设计` 或 `/制定计划`）

## 注意事项

1. **PRD 与 Spec 的关系**：
   - Spec 侧重 What & Why（要做什么、为什么做、如何验收）
   - PRD 侧重 How（业务层面：业务流程、交互设计、字段规则）
   - PRD 应基于 Spec 的需求生成，确保业务需求一致

2. **图表生成**：
   - **功能架构图**：⭐ 强烈推荐使用 PlantUML（支持长线样式、颜色分组）
   - **模块调用关系图**：⭐ 强烈推荐使用 PlantUML（支持不同线型区分调用方式）
   - **流程图**：推荐使用 PlantUML 或 Mermaid
   - 所有图表必须包含在 PRD 文档中

3. **字段验证规则**：
   - 使用标准验证规则（参考 `.cursor/commands/编写规格.md` 中的验证规则）
   - 手机号：`^1[3-9]\d{9}$`
   - 邮箱：`^[a-zA-Z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}$`
   - 密码：`^(?=.*[A-Za-z])(?=.*\d)[A-Za-z\d!@#$%^&*]{8,20}$`

4. **操作风险分级**：
   - 根据 `docs/rules/prototyping/page-design-rules.md` 的操作分级规范
   - 在 PRD 中标注每个操作的风险等级和确认机制

5. **版本管理**：
   - PRD 版本号从 v1.0 开始
   - 每次重大变更需要更新版本号
   - 在变更记录中记录更新历史
